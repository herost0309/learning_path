# ADR-001: 通用平台应用使用 DB 存储 Metadata

| 字段 | 内容 |
|------|------|
| **状态** | 提议（Proposed） |
| **日期** | 2026-06-09 |
| **决策者** | 架构评审委员会 |
| **范围** | 通用平台应用整体架构 |

---

## 1. 背景（Context）

我们正在构建一个基于 AWS 的**通用平台应用**，核心能力包括：
- **文档流转与审批**：文档的上传、存储、流转、审批全生命周期管理
- **工作流编排引擎**：支持多种业务场景的流程编排与任务调度

技术选型为 AWS 云原生方案，使用 **S3 存储文档实体**，需要确定 **metadata（元数据）的存储策略**。

### 当前争议

架构师认为：应用定位是"通道"（Channel），不应引入自有数据库存储 metadata。metadata 的持久化应由上下游系统负责，本应用仅通过 API 连接获取。

我们的观点：作为文档流转 + 工作流编排的通用平台，**必须有自有 DB 存储 metadata**，否则核心功能无法实现。

---

## 2. 决策（Decision）

**通用平台应用引入自有数据库（PostgreSQL / Aurora）存储 metadata。**

这不是一个技术选型问题（PG vs 其他），而是一个**应用定位问题**：我们的应用是否属于"无状态通道"，还是"有状态的通用平台"。

---

## 3. 理由（Rationale）

### 3.1 应用能力边界决定存储需求，而非标签决定

应用的存储需求不应由"通道"这个标签决定，而应由**应用实际承担的能力边界**决定。

> 📌 TOGAF 架构原则：架构决策应基于能力和需求，而非先验假设。

我们的应用提供两类核心能力，每一类都**内在要求持久化存储**：

| 核心能力 | 为什么必须有 DB |
|----------|---------------|
| **文档流转/审批** | 需要维护文档目录索引（S3 没有目录/搜索能力）、文档与流程的关联关系、版本追踪 |
| **工作流编排引擎** | 必须追踪每个流程实例的运行时状态：当前步骤、任务分配、审批结果、超时计时 |

**关键问题**：一个工作流引擎在不知道"流程走到第几步、谁在审批"的情况下，如何运转？答案是——不可能。这不是设计偏好，是功能必然。

### 3.2 四类 Metadata 的数据主权分析

我们需要存储的 4 类 metadata，逐一分析其数据主权归属：

| Metadata 类型 | 谁产生的？ | 谁消费的？ | 能否放到上下游？ | 结论 |
|---|---|---|---|---|
| 文档索引/目录 | **本平台**（文档上传时创建） | 本平台查询 | ❌ 上下游不知道我们的文档组织方式 | **必须自存** |
| 流程/任务状态 | **本平台**（流程引擎运行时产生） | 本平台调度 | ❌ 上下游都不是流程的执行者 | **必须自存** |
| 配置/映射规则 | **平台管理员**配置 | 本平台执行 | ❌ 这是平台配置，非业务数据 | **必须自存** |
| 租户/权限信息 | **本平台**（多租户管理） | 本平台鉴权 | ❌ 权限是平台基础能力 | **必须自存** |

**结论：4 类 metadata 全部由本平台产生和消费，无一适合放到上下游系统。**

### 3.3 "通道"定位的边界再定义

我们认同架构师"通道"理念中合理的部分——**本平台不是业务数据（如订单、客户、合同）的最终归属**。

但"通道"和"没有数据库"不能画等号。真实世界中，每一类"通道"系统都有自己的运行数据：

| "通道"系统 | 传输的业务数据 | 自身必须存储的运行数据 |
|-----------|--------------|---------------------|
| 快递公司 | 包裹里的货物 | 运单号、当前位置、配送人、签收状态 |
| API 网关 | 业务请求/响应 | 路由规则、限流配置、访问日志 |
| 消息队列 | 消息内容 | 消费偏移量、分区分配、死信队列 |
| **我们的平台** | 文档内容（S3） | 流程状态、审批记录、目录索引、权限配置 |

> 💡 **类比**：快递公司是"通道"——它不拥有包裹里的货物，但它必须有运单追踪系统。我们的平台是文档流转的"通道"——不拥有业务数据，但必须有流程状态追踪系统。

### 3.4 上下游系统现实约束

当前上下游系统**技术栈参差不齐**：
- 部分系统成熟，API 稳定
- 部分系统老旧或 API 不完善
- 部分系统还在规划中，API 能力未确定

如果把 metadata 存储责任推给上下游，将面临：
- **可用性不可控**：平台可用性 = min(所有上下游可用性)
- **部署依赖链**：新区域部署必须等待所有上下游就绪
- **无法独立演进**：每次上下游 API 变更都影响平台核心功能
- **通用性丧失**：无法快速对接新的业务场景

### 3.5 通用平台的多场景支撑要求

作为**通用平台**，需要支撑多种业务场景，这意味着：
- 不同场景的上下游系统不同，metadata schema 也不同
- 如果 metadata 放在各自的上下游，平台需要适配 N 套不同的 API 和数据模型
- 自有 DB 可以**统一 metadata 模型**，屏蔽上下游差异，降低对接成本

### 3.6 业界架构原则支撑

**TOGAF 微服务数据主权原则（G246）**：
> *"A microservice must be the authoritative source of the data managed by its services."*
> （微服务必须是其服务所管理数据的权威来源）

**TOGAF 元数据管理能力域（G234）**：
元数据管理被定义为一个独立的能力域，包含 Metadata Repository Design（元数据仓库设计），需要独立的存储和治理。

**TOGAF 微服务自包含原则（G21I）**：
微服务应具备 Self-containment（自包含）和 Independence（独立性），依赖外部系统存储运行时数据会丧失这些特性。

---

## 4. 替代方案（Alternatives Considered）

### 方案 B：不引入自有 DB，metadata 全部由上下游管理，平台通过 API 获取

| 评估维度 | 评分 | 说明 |
|----------|------|------|
| 可行性 | ❌ 不可行 | 流程状态、配置规则等数据上下游无法提供 |
| 可靠性 | ❌ 差 | 依赖参差不齐的上下游，整体可用性下降 |
| 部署效率 | ❌ 差 | 每个新区域需等待所有上下游就绪 |
| 开发效率 | ❌ 差 | 需适配 N 套 API，额外处理认证/重试/熔断 |
| 通用性 | ❌ 差 | 无法统一 metadata 模型，每对接一个新场景都要定制 |
| 运维复杂度 | ❌ 高 | 故障排查需要跨系统排查 |

**结论：方案 B 在技术上不可行，在工程实践上代价过高。**

### 方案 C：引入自有 DB + 与上下游保持数据同步（折中方案）

| 评估维度 | 评分 | 说明 |
|----------|------|------|
| 可行性 | ✅ 可行 | 核心数据自存，业务数据通过事件同步 |
| 可靠性 | ✅ 好 | 核心功能不依赖外部，业务数据有最终一致性 |
| 部署效率 | ✅ 好 | 核心部署包自包含 |
| 通用性 | ✅ 好 | 统一 metadata 模型 + 灵活对接 |
| 复杂度 | ⚠️ 中等 | 需要维护事件同步机制 |

**结论：方案 C 是推荐的折中方案，在满足功能需求的同时，保留了与上下游协作的灵活性。**

---

## 5. 后果（Consequences）

### 引入自有 DB 后

**收益**：
- 工作流引擎可以正常运转（状态持久化）
- 文档检索有性能保障（本地索引查询 < 10ms）
- 多区域部署标准化（PG + S3 + App = 完整部署包）
- 通用平台可以统一 metadata 模型，快速对接新场景
- 开发效率提升（标准 ORM/SQL，无大量 API 适配代码）

**成本**：
- PG/Aurora 实例费用（Aurora Serverless 约 $50-100/月/区域）
- 需要数据库运维能力（schema 迁移、备份恢复、监控）
- 需要与上下游的数据同步机制（如果采用方案 C）

**风险缓解**：
- 使用 Aurora Serverless 降低运维负担
- 使用 Flyway/Liquibase 管理 schema 迁移
- S3 文档存储 + PG metadata 存储的分层架构，文档实体不占 DB 空间

## 6. 关键反问（用于架构评审讨论）

> **Q1**: "如果我们是纯通道，那么一个文档在审批流程中走到第3步、等待张三审批——这个状态应该存在哪里？"
>
> **Q2**: "如果 metadata 放在上下游，当上下游系统 A 挂掉时，平台的文档流转功能还能正常工作吗？"
>
> **Q3**: "作为通用平台，对接 N 个不同的业务场景时，metadata 放在各场景的上下游系统里，平台如何统一管理和检索？"
