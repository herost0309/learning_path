### Slide 3: 背景与问题定义

**要点**:
- 我们正在构建一个基于 AWS 的**通用平台应用**
- 核心能力：文档流转/审批 + 工作流编排引擎
- 技术方案：S3 存储文档实体 + PG 存储元数据
- **当前争议**：架构师认为应用定位是"通道"，不应引入自有 DB

**核心问题（高亮）**:
> 我们的应用是否应引入自有 DB 存储 metadata？
> 这不是一个技术选型问题，而是一个**应用定位问题**。

---

### Slide 4: 应用能力边界分析

**要点**: 先明确应用实际做什么，再决定需要什么基础设施

| 能力 | 描述 | 是否需要持久化 |
|------|------|---------------|
| 文档流转/审批 | 文档上传、流转、审批全生命周期 | ✅ 文档索引、版本、关联关系 |
| 工作流编排引擎 | 多场景流程编排、任务调度、状态机 | ✅ 流程状态、任务分配、审批记录 |

**关键问题**:
> 工作流引擎在不知道"流程走到第几步、谁在审批"的情况下，如何运转？

### Slide 5: Metadata 归属分析

**要点**: 逐一分析 4 类 metadata 的数据主权

**展示归属分析表**:

| Metadata 类型 | 产生者 | 能否放上下游 | 结论 |
|---|---|---|---|
| 文档索引/目录 | 本平台 | ❌ | 必须自存 |
| 流程/任务状态 | 本平台 | ❌ | 必须自存 |
| 配置/映射规则 | 平台管理员 | ❌ | 必须自存 |
| 租户/权限信息 | 本平台 | ❌ | 必须自存 |

**结论**: 4 类 metadata **全部由本平台产生和消费**，无一适合放到上下游。

---

### Slide 6: "通道"定位的边界再定义

**要点**: 我们认同"通道"理念中合理的部分，但需要明确边界

**类比表格**:

| "通道"系统 | 传输的业务数据 | 自身必须存储的运行数据 |
|-----------|--------------|---------------------|
| 快递公司 | 包裹里的货物 | 运单号、当前位置、配送人、签收状态 |
| API 网关 | 业务请求/响应 | 路由规则、限流配置、访问日志 |
| **我们的平台** | 文档内容 | 流程状态、审批记录、目录索引、权限配置 |

**关键话术**:
> "我们认同不是业务数据的 owner。但快递公司虽然是通道，它必须有运单追踪系统。
> 这不是货物数据，而是通道自身的运行数据——我们的平台也一样。"

### Slide 7: 上下游系统现实约束

**要点**: 上下游参差不齐是客观现实

- 部分系统成熟，API 稳定
- 部分系统老旧，API 不完善
- 部分系统还在规划中

**如果无 DB 依赖上下游 API**:

| 风险 | 影响 |
|------|------|
| 上游挂了 | 整个审批流程瘫痪 |
| API 响应慢 | 用户体验极差（50-5000ms vs <10ms） |
| API 格式变更 | 频繁适配，维护成本高 |
| 新区域部署 | 必须等所有上下游就绪，周期翻 10 倍 |

### Slide 9: 业界架构原则支撑

**要点**: 不是我们的个人观点，是业界共识

**三条原则**:
1. **TOGAF G246 — 微服务数据主权**
   > "微服务必须是其服务所管理数据的权威来源"
   → 我们的流程状态、文档索引是我们自己管理的数据

2. **TOGAF G234 — 元数据管理是独立能力域**
   > 元数据管理需要自己的 Repository Design
   → 不是 API 转发能替代的

3. **TOGAF G21I — 微服务自包含原则**
   > 微服务应具备自包含性和独立性
   → 依赖外部存储运行时数据会丧失这些特性

### Slide 10: 推荐方案

**要点**: 提出建设性的折中方案

**方案 C: 核心数据自存 + 业务数据事件同步**

```
平台自存（不可妥协）           与上下游同步（体现通道精神）
─────────────────           ──────────────────
• 流程实例状态                • 业务分类（EventBridge）
• 任务分配/审批记录            • 关联业务ID（SQS事件）
• 文档索引/目录               • 业务状态变更（事件刷新）
• 配置/映射规则
• 租户/权限信息
```

**好处**:
1. 承认了"通道"理念中合理的部分（不拥有业务数据）
2. 坚守了必须自存的底线（工作流引擎的状态数据）
3. 展示了积极寻找双方都能接受的方案

### Slide 11: 成本估算

**要点**: 有 DB 的成本是可控的

| 成本项 | 金额 | 说明 |
|--------|------|------|
| Aurora Serverless | ~$50-100/月/区域 | 按使用量计费，闲置时极低 |

### Slide 12: 总结与决策请求

**三条核心论证**:

1. **功能必然性**: 工作流引擎没有 DB 就无法工作——这不是设计偏好，是功能必然
2. **数据主权**: 4 类 metadata 全部由平台产生，上下游无法提供
3. **通用平台需求**: 多场景支撑 + 多区域部署 = 必须有独立的标准化存储

**决策请求**:
> 请求架构评审委员会批准：通用平台引入自有 DB（PG/Aurora）存储 metadata，
> 采用方案 C（核心自存 + 业务同步）的折中策略。

### Slide 13: Q&A

**预设问题与回答准备**:

**Q: 为什么不直接用 S3 存 metadata？**
> S3 没有关系查询能力（JOIN、条件筛选、聚合），无法支持工作流引擎的状态查询、文档的目录检索、权限的条件匹配等需求。

**Q: DB 会不会成为性能瓶颈？**
> Aurora Serverless 自动扩缩容，支持读副本。metadata 是轻量数据（文本/JSON），不存文档实体（S3），单区域可轻松支撑万级 QPS。

**Q: 多区域数据同步怎么处理？**
> 每个区域独立 PG 实例，不跨区域同步。业务数据同步通过 EventBridge/SQS 事件驱动，最终一致性。

**Q: 为什么不让工作流引擎也做无状态的？**
> 工作流引擎的本质是状态机管理（BPMN/状态机模型），无状态的工作流引擎在学术和工程上都是矛盾的。业界所有工作流引擎（Camunda、Temporal、AWS Step Functions）都有自己的持久化层。

### 备用 Slide: AWS Step Functions 类比

**要点**: AWS 自己的工作流服务也用 DB

- AWS Step Functions 本质就是工作流引擎
- 它内部使用 DynamoDB 存储执行历史和状态
- 即使是 AWS 原生的"通道型"服务，也需要持久化层
- 我们的方案和 AWS 自身的设计理念一致
