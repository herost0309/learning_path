# Skill: 代码考古

> 用途：对未知或半已知代码仓进行系统性逆向分析，从代码中提取业务逻辑、架构设计和实现细节，补齐缺失的技术文档。
> 适用场景：接手遗留系统、团队人员流失导致知识断层、代码仓缺少文档需要补齐。

## 触发条件

- 接手一个没有文档的遗留代码仓
- 团队核心成员离职，需要快速理解其负责的模块
- 需要为现有代码仓补齐完整的技术文档
- 代码仓文档严重过时，需要重新梳理
- 跨团队协作时需要快速理解对方的系统

## 三阶段方法概览

```
Phase 1: 全局扫描    →  产出项目全景图
Phase 2: 模块深潜    →  逐模块提取业务规则和数据流
Phase 3: 知识产出    →  生成可维护的完整文档集
```

---

## Phase 1: 全局扫描

**目标**：用最少的时间建立对整个代码仓的全局认知。

### 1.1 项目全景图

```
Prompt：
分析当前项目的完整代码结构，生成项目全景图。

要求：
1. 识别技术栈（框架、语言版本、构建工具、主要依赖）
2. 列出目录结构及其职责说明
3. 列出所有业务模块（从目录名和代码推断）
4. 识别项目类型（单体/微服务/Serverless/类库）
5. 估算代码规模（文件数、代码行数的大致量级）

输出格式：Markdown，包含目录树和技术栈表格
```

**产出模板**：

```markdown
# [项目名] 全景图

## 技术栈
| 类别 | 技术 | 版本 | 用途 |
|------|------|------|------|
| 语言 | ... | ... | ... |
| 框架 | ... | ... | ... |
| 数据库 | ... | ... | ... |
| 消息队列 | ... | ... | ... |

## 目录结构
```
[项目根]/
├── [目录] -- [职责说明]
└── ...
```

## 模块清单
| 模块名 | 路径 | 职责推断 | 代码量估算 |
|--------|------|---------|-----------|
```

### 1.2 入口点地图

```
Prompt：
扫描项目代码，找出所有"入口点"——即外部触发系统执行的位置。

需要找出的入口类型：
1. HTTP API 端点（路由定义、Controller）
2. 定时任务（Cron Job、Scheduled Task）
3. 消息消费者（MQ Listener、Event Handler）
4. CLI 命令（命令行入口）
5. 外部触发器（Webhook、Callback）
6. 主程序入口（main 函数、启动脚本）

对每个入口点列出：
- 入口类型和路径
- 对应的代码文件和行号
- 简要功能描述（从代码推断）

输出格式：按类型分组的表格
```

**产出模板**：

```markdown
# [项目名] 入口点地图

## HTTP API 端点
| 方法 | 路径 | 文件 | 功能 |
|------|------|------|------|
| GET | /api/users | src/controller/UserController.java:42 | 用户列表查询 |

## 定时任务
| 调度规则 | 文件 | 功能 |
|----------|------|------|
| 0 2 * * * | src/job/DataSyncJob.java | 每日凌晨同步数据 |

## 消息消费者
| Topic/Queue | 文件 | 功能 |
|-------------|------|------|
| order.created | src/listener/OrderListener.java | 处理订单创建事件 |
```

### 1.3 依赖关系图

```
Prompt：
分析项目的模块间依赖关系和外部服务依赖。

要求：
1. 模块间调用关系：哪个模块调用了哪个模块
2. 外部服务依赖：调用了哪些第三方 API 或服务
3. 数据库依赖：使用了哪些数据库、缓存、消息队列
4. 共享依赖：哪些模块共享了公共组件

输出格式：
- 用文字描述依赖关系（A 依赖 B）
- 标注依赖方向（单向/双向）
- 标注依赖类型（同步调用/异步消息/共享数据）
```

### 1.4 现有文档盘点

```
Prompt：
扫描项目中所有已存在的文档文件，包括：
1. README.md 及其他 Markdown 文件
2. docs/ 目录下的所有文件
3. 代码中的注释（特别是文件头注释、模块级注释）
4. Swagger/OpenAPI 定义文件
5. 配置文件中的注释
6. CHANGELOG、CONTRIBUTING 等社区文档

对每份文档评估：
- 内容概述
- 最后更新时间（从 git log 获取）
- 与当前代码的一致性（是否过时）
- 缺失的关键信息

输出格式：表格，按文档质量排序
```

## Phase 2: 模块深潜

**目标**：对每个业务模块逐一深入分析，提取业务规则和数据流。

### 2.1 代码与文档交叉比对

对每个模块执行：

```
Prompt：
分析 [模块路径] 目录下的代码，并与以下已有文档交叉比对：
[粘贴已有文档内容，或注明"无已有文档"]

要求：
1. 总结模块实际功能（从代码推断）
2. 与已有文档的差异点（如果文档存在）
3. 文档中描述但代码中不存在的功能（已删除？）
4. 代码中存在但文档中未提及的功能（文档缺失）
5. 标注每个差异的严重程度（关键/中等/低）

输出格式：
- 差异列表，每条包含：位置、描述、严重程度
```

### 2.2 提取业务规则

```
Prompt：
深度分析 [模块路径] 目录下的代码，提取其中隐含的业务规则。

重点关注：
1. if/else 条件分支 —— 每个分支代表什么业务场景？
2. 枚举值和常量 —— 它们的业务含义是什么？
3. 状态机逻辑 —— 实体有哪些状态、状态间如何流转？
4. 校验规则 —— 哪些输入被拒绝、原因是什么？
5. 权限检查 —— 哪些操作需要什么角色/权限？
6. 异常处理 —— 哪些异常场景被处理、如何处理？
7. 计算逻辑 —— 金额、折扣、评分等业务计算公式

对每条业务规则标注置信度：
- [高] 规则明确，代码中有清晰的注释或命名
- [中] 规则可推断，但需要人工确认
- [低] 规则不确定，AI 无法确定业务含义

输出格式：决策表 + 置信度标注
```

**产出模板**：

```markdown
# [模块名] 业务规则

## 状态流转
| 当前状态 | 触发条件 | 目标状态 | 置信度 |
|----------|---------|---------|--------|
| 待支付 | 支付成功回调 | 已支付 | [高] |
| 已支付 | 超过7天未发货 | 自动取消 | [中] |

## 校验规则
| 校验项 | 规则 | 错误信息 | 置信度 |
|--------|------|---------|--------|
| 手机号 | 必须为11位数字 | "手机号格式错误" | [高] |
| 订单金额 | > 0 且 <= 999999 | "金额超出范围" | [高] |

## 计算逻辑
| 计算项 | 公式/规则 | 置信度 |
|--------|---------|--------|
| 折扣 | VIP用户95折，SVIP用户9折 | [中] |
```

### 2.3 生成数据流图

```
Prompt：
追踪 [模块路径] 中的数据流，从入口到存储到响应的完整路径。

对每个主要功能（API 端点/定时任务/消息处理），画出：
1. 请求从哪里进来
2. 经过哪些处理步骤（函数调用链）
3. 读写了哪些数据表/缓存/外部服务
4. 响应什么数据
5. 异常情况如何处理

输出格式：用文字描述调用链，并标注每个步骤的代码位置（文件:行号）
```

**产出模板**：

```markdown
# [模块名] 数据流

## POST /api/orders（创建订单）
```
请求入口: OrderController.create() [src/controller/OrderController.java:35]
  → 参数校验: OrderValidator.validate() [src/validator/OrderValidator.java:12]
  → 查询商品: ProductService.getById() [src/service/ProductService.java:88]
  → 计算价格: PricingEngine.calculate() [src/pricing/PricingEngine.java:23]
  → 检查库存: InventoryService.check() [外部服务: inventory-api]
  → 创建订单: OrderRepository.save() [写入: orders 表]
  → 发送通知: NotificationService.send() [异步: MQ order.created]
  → 返回: OrderResponse [200 OK]
```

异常路径：
- 商品不存在 → 404
- 库存不足 → 409
- 价格计算失败 → 500
```

### 2.4 标注不确定项

```
Prompt：
在分析 [模块路径] 的代码后，列出你无法确定的部分。

包括但不限于：
1. 不确定业务含义的变量名/函数名
2. 不确定是否还在使用的旧代码
3. 不确定是否有副作用的全局状态修改
4. 不确定调用链是否完整的异步逻辑
5. 不确定是否为 bug 的可疑逻辑

每条不确定项需标注：
- 位置（文件:行号）
- 不确定原因
- 建议的确认方式（问谁、查什么）
```

## Phase 3: 知识产出

**目标**：将分析结果转化为可维护的文档集。

### 3.1 模块架构文档

```
Prompt：
基于对 [模块路径] 的完整分析，生成模块架构文档。

要求：
1. 模块职责（一句话）
2. 对外接口列表
3. 内部组件图（用 Mermaid 语法）
4. 依赖关系
5. 配置项说明
6. 已知问题和风险

输出到: docs/architecture/[模块名].md
```

**产出模板**：

```markdown
# [模块名] 模块架构

## 职责
[一句话描述]

## 组件结构
​```mermaid
graph TD
    A[Controller] --> B[Service]
    B --> C[Repository]
    B --> D[External API]
    B --> E[Cache]
​```

## 对外接口
| 接口 | 方法 | 描述 |
|------|------|------|

## 依赖关系
| 依赖项 | 类型 | 用途 |
|--------|------|------|

## 配置项
| 配置键 | 默认值 | 说明 |
|--------|--------|------|

## 已知风险
| 风险 | 严重度 | 说明 |
|------|--------|------|
```

### 3.2 数据模型文档

```
Prompt：
分析项目中所有数据模型定义（Entity/Model/Schema/Migration），生成数据模型文档。

要求：
1. 每张表的字段列表（字段名、类型、约束、含义）
2. 表间关系（一对多、多对多、外键）
3. 索引设计
4. 枚举字段的值域和业务含义
5. 数据生命周期（何时创建、何时更新、何时删除）

输出到: docs/architecture/data-model.md
```

### 3.3 API 文档

```
Prompt：
从路由/Controller 定义中提取所有 API 端点，生成 API 文档。

要求：
1. 按模块分组
2. 每个端点包含：方法、路径、描述、参数、响应、错误码
3. 标注认证要求
4. 标注是否为分页接口
5. 标注废弃的接口（如有 @Deprecated 注解或类似标记）

输出到: docs/api/[模块名].md
```

### 3.4 风险地图

```
Prompt：
基于对整个项目的分析，生成风险地图。

风险分类：
1. 高复杂度代码（圈复杂度高的函数）
2. 缺少测试的核心逻辑
3. 硬编码的配置值
4. 已知的 TODO/FIXME/HACK 注释
5. 频繁变更的热点文件（从 git log 分析）
6. 依赖过时或废弃的外部库
7. 潜在的安全问题（SQL 拼接、未校验输入等）

输出到: docs/architecture/risk-map.md
```

## 完整示例：分析订单模块

以下是一个完整的代码考古示例，展示如何分析一个典型的订单模块。

### Step 1: 全局扫描订单模块

```
Prompt：
分析 src/modules/order/ 目录，完成以下任务：

1. 列出所有文件及其大致职责
2. 找出所有入口点（API 端点、事件监听、定时任务）
3. 列出 order 模块依赖的其他模块和外部服务
4. 找出数据模型定义
```

### Step 2: 深入分析订单状态流转

```
Prompt：
在 src/modules/order/ 目录中，找到订单状态相关的所有代码。

重点分析：
1. 订单有哪些状态？从枚举/常量中提取
2. 状态之间的流转规则是什么？
3. 每次状态变更会触发什么操作？
4. 有没有异常状态的处理逻辑？

将分析结果整理为状态流转表。
```

### Step 3: 追踪订单创建的完整数据流

```
Prompt：
追踪"创建订单"功能从入口到存储的完整调用链。

从 OrderController.create() 开始，逐步追踪：
1. Controller 接收什么参数
2. Service 层做了什么处理
3. 调用了哪些外部服务
4. 写入了哪些数据表
5. 返回什么响应
6. 有哪些异常分支

在每一步标注代码位置（文件名:行号）。
```

### Step 4: 生成订单模块文档

```
Prompt：
基于以上对订单模块的分析，生成以下文档：

1. 模块架构文档 → docs/architecture/order-module.md
2. 订单 API 文档 → docs/api/order-api.md
3. 订单数据模型 → docs/architecture/data-model.md 的订单部分
4. 订单业务规则 → docs/architecture/order-business-rules.md

每份文档都按照置信度体系标注：
- 确定的内容标注为 [高置信]
- 推断的内容标注为 [中置信]，并说明推断依据
- 不确定的内容标注为 [低置信]，并建议确认方式
```

## 使用建议

### 工作节奏

| 阶段 | 适用规模 | 建议方式 |
|------|---------|---------|
| Phase 1 全局扫描 | 任意规模 | 一次性完成，产出全景图 |
| Phase 2 模块深潜 | 按模块拆分 | 每次聚焦 1 个模块，优先级从高到低 |
| Phase 3 知识产出 | 伴随 Phase 2 | 每个模块分析完就产出文档 |

### 优先级排序

模块分析的优先级建议：

1. **核心业务模块**（直接影响收入的模块）
2. **高频变更模块**（从 git log 统计最近修改频率）
3. **跨团队依赖模块**（被其他团队调用的接口）
4. **问题频发模块**（Bug 高发区域）
5. **即将重构模块**（有技术改造计划的模块）

### 与 knowledge-sync Skill 配合

代码考古完成后，使用 `knowledge-sync` Skill 建立长期同步机制，确保文档与代码持续一致。

### 与 generate-docs Skill 配合

代码考古侧重"逆向理解代码"，generate-docs 侧重"从代码生成文档"。两者配合使用：
- 先用代码考古理解代码
- 再用 generate-docs 生成标准化文档
