# Skill: 发布版本 / 部署

> 用途：标准化的发布和部署流程，减少人为错误和上线焦虑。

## 触发条件

- 准备发布新版本
- 需要部署到测试/预发/生产环境

## 输入要求

- **版本号**: 遵循语义化版本（SemVer）
- **变更范围**: 本次发布包含的主要变更
- **目标环境**: 测试 / 预发 / 生产
- **回滚方案**: 如果出问题如何回滚

## 执行步骤

### Step 1: 发布前检查

```markdown
## 发布检查清单

### 代码检查
- [ ] 所有 MR 已合并到发布分支
- [ ] 代码冻结：不再接受新的 MR
- [ ] 无未解决的安全漏洞（依赖扫描通过）

### 测试检查
- [ ] 单元测试全部通过
- [ ] 集成测试全部通过
- [ ] 回归测试覆盖核心场景
- [ ] 性能测试（如涉及性能相关改动）

### 配置检查
- [ ] 环境变量已更新（如需要）
- [ ] 数据库迁移脚本已准备（如需要）
- [ ] 功能开关配置正确（如使用）

### 文档检查
- [ ] CHANGELOG 已更新
- [ ] API 文档已更新（如涉及接口变更）
- [ ] 运维通知已发送（如需要）
```

### Step 2: 版本号管理

根据变更类型确定版本号：

- **PATCH (x.y.Z)**: Bug 修复，无新功能
- **MINOR (x.Y.z)**: 新功能，向后兼容
- **MAJOR (X.y.z)**: 破坏性变更

### Step 3: 构建与打包

```bash
# 按项目实际情况调整
# 1. 拉取最新代码
# 2. 安装依赖
# 3. 运行测试
# 4. 构建
# 5. 打包 Docker 镜像（如使用）
# 6. 推送到制品库
```

### Step 4: 部署流程

```
测试环境 → 预发环境 → 生产环境
  ↓            ↓           ↓
自动部署    自动部署     手动确认后部署
```

每个环境部署后：
1. 冒烟测试通过
2. 核心功能验证
3. 日志监控无异常

### Step 5: 发布后验证

- [ ] 监控面板无异常告警
- [ ] 核心业务流程验证通过
- [ ] 日志无 Error 级别异常（预期内的除外）
- [ ] 性能指标在正常范围内
- [ ] 观察 15-30 分钟确认稳定

### Step 6: 收尾

- 更新 CHANGELOG
- 通知相关团队
- 归档发布分支（如使用）
- 记录发布经验教训

## 回滚预案

如果发布后发现问题：

1. **L1 - 配置回滚**: 关闭功能开关（最快，秒级）
2. **L2 - 版本回滚**: 回退到上一版本镜像（分钟级）
3. **L3 - 数据回滚**: 执行数据回滚脚本（需谨慎评估）

## 注意事项

- 生产环境发布建议在工作日进行，避免周五发布
- 重大变更采用灰度发布策略
- 保持回滚方案随时可用
