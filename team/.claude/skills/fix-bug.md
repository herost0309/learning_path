# Skill: 修复 Bug

> 用途：标准化的 Bug 修复流程，确保根因分析和回归测试。

## 触发条件

当收到 Bug 报告或发现线上问题时使用此 Skill。

## 输入要求

- **Bug 描述**: 现象是什么
- **复现步骤**: 如何重现
- **期望行为**: 正确应该怎样
- **错误信息**: 日志/报错（如有）

## 执行步骤

### Step 1: 复现与分析

1. 根据复现步骤尝试复现 Bug
2. 分析相关代码，定位问题根因
3. 检查是否有关联的已知问题

**输出**：Bug 根因分析（写在代码注释或 MR 描述中）

### Step 2: 编写失败测试

在修复前，**先写一个能复现 Bug 的测试用例**：

- 测试名称格式：`test_[功能]_should_[期望行为]_when_[条件]`
- 测试必须能稳定复现 Bug
- 这个测试是回归保护网

### Step 3: 实施修复

1. **最小改动原则**: 只修改修复 Bug 所需的代码，不做额外重构
2. 修复后确认失败测试通过
3. 检查修复是否影响其他功能（运行相关测试套件）

### Step 4: 影响评估

- [ ] 修复是否影响其他模块？
- [ ] 是否需要数据迁移/修复？
- [ ] 是否需要更新文档？
- [ ] 是否需要通知其他团队？

### Step 5: 提交

提交信息格式：`fix: [简要描述问题] (#issue编号)`

MR 描述中包含：
- **根因**: 为什么会出现这个 Bug
- **修复方案**: 怎么修的
- **测试**: 添加了什么测试防止回归
- **影响范围**: 可能影响的模块

## 注意事项

- 不要在 Bug 修复 PR 中混入重构或其他改动
- 如果发现 Bug 暴露了更深层的架构问题，记录到 `docs/specs/` 作为后续重构任务
- 线上紧急修复可以跳过部分步骤，但事后必须补齐测试
