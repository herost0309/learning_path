# Skill: AI 生成技术文档

> 用途：用 AI 自动从代码生成各类技术文档，解决"没人写文档"的问题。
> 这是少数"零争议"的 AI 应用场景——没人喜欢写文档，但人人都需要文档。

## 触发条件

- 新功能开发完成，需要补充文档
- 旧模块缺少文档，需要补齐
- API 接口需要生成/更新文档
- 需要生成架构说明文档
- 代码变更后文档需要同步更新
- 需要往 Confluence 发布文档

## 输入要求

- **文档类型**: API 文档 / 架构文档 / 模块说明 / 运维手册 / 变更日志
- **目标范围**: 需要生成文档的代码路径或模块
- **输出位置**: 本地 Markdown / Confluence 页面 / 代码注释
- **受众**: 新人 / 团队内部 / 外部合作方

## 执行步骤

### Step 1: 确定文档类型和模板

根据需求选择合适的文档模板：

| 文档类型 | 模板位置 | 典型受众 |
|----------|----------|----------|
| API 接口文档 | `docs/api/` | 前端、测试、外部合作方 |
| 模块架构文档 | `docs/architecture/` | 团队新人、跨团队协作 |
| 变更日志 | `CHANGELOG.md` | 全团队 |
| 运维手册 | `docs/ops/` | 运维、值班人员 |
| Spec 文档 | `docs/specs/` | 开发者、产品经理 |

### Step 2: AI 分析代码

根据文档类型，给 AI 不同的分析指令：

#### 生成 API 文档

```
Prompt：
分析以下路由/Controller 文件，生成 API 接口文档。

要求：
1. 列出所有 HTTP 端点（方法、路径、描述）
2. 列出每个端点的请求参数（路径参数、查询参数、请求体）
3. 列出响应格式（成功响应、错误响应）
4. 标注认证要求
5. 标注是否为分页接口

文件路径：[粘贴路径]
```

#### 生成模块架构文档

```
Prompt：
分析 src/modules/[模块名]/ 目录，生成模块架构文档。

要求：
1. 用一句话描述这个模块的职责
2. 列出模块对外的接口（函数/类/API）
3. 画出数据流（从输入到输出经过哪些组件）
4. 列出模块依赖的外部服务/数据库
5. 标注高复杂度或高风险的函数
6. 列出建议的优化方向

输出格式：Markdown
```

#### 生成变更日志

```
Prompt：
根据以下 git commit 历史，生成 CHANGELOG 条目。

规则：
- feat: 归入 "Features" 分类
- fix: 归入 "Bug Fixes" 分类
- refactor: 归入 "Refactoring" 分类
- breaking change: 单独标注为 "BREAKING CHANGES"
- 每个 commit 写一句人话描述（不要直接抄 commit message）
- 按时间倒序排列

commit 历史：
[粘贴 git log 输出]
```

### Step 3: 人工审查

AI 生成的文档**必须审查**以下方面：

- [ ] **准确性**：AI 是否正确理解了代码逻辑？
- [ ] **完整性**：是否有遗漏的接口/参数/场景？
- [ ] **时效性**：是否反映了代码的最新状态？
- [ ] **语言质量**：描述是否清晰、无歧义？
- [ ] **安全性**：是否泄露了敏感信息（密钥、内部 IP 等）？

> AI 最常见的文档问题：描述正确但遗漏了边界情况；生成的参数列表少了可选参数。

### Step 4: 发布到 Confluence

#### 方法 A：手动粘贴（最简单）

1. AI 生成 Markdown 格式文档
2. 在 Confluence 新建页面
3. 粘贴 Markdown 内容（Confluence 自动转为富文本）
4. 设置页面标签（如 `api-doc`、`auto-generated`）

#### 方法 B：使用脚本自动发布

```python
# 用 AI 生成这个脚本，填入你的 Confluence 信息即可使用
# pip install requests markdown

import requests
import markdown

def publish_to_confluence(title, markdown_content, space_key, parent_id=None):
    """
    将 Markdown 文档发布到 Confluence
    """
    html_content = markdown.markdown(markdown_content)

    url = "https://your-company.atlassian.net/wiki/rest/api/content"
    headers = {
        "Authorization": "Bearer YOUR_API_TOKEN",
        "Content-Type": "application/json"
    }

    payload = {
        "type": "page",
        "title": title,
        "space": {"key": space_key},
        "body": {
            "storage": {
                "value": html_content,
                "representation": "storage"
            }
        }
    }

    if parent_id:
        payload["ancestors"] = [{"id": parent_id}]

    response = requests.post(url, json=payload, headers=headers)
    return response.json()

# 使用示例
publish_to_confluence(
    title="订单模块 API 文档",
    markdown_content=open("docs/api/orders.md").read(),
    space_key="DEV"
)
```

```
"请分析 src/modules/orders/ 目录的代码，
 生成 API 文档并发布到 Confluence 的 DEV 空间，

```

### Step 5: 文档更新策略

| 场景 | 更新方式 | 触发条件 |
|------|---------|---------|
| 新接口上线 | AI 重新扫描路由文件，增量更新 | 每次发版 |
| 接口参数变更 | AI 对比变更，更新文档 | MR 合并时 |
| 模块重构 | AI 重新分析模块，覆盖更新 | 重构完成后 |
| 定期巡检 | AI 对比代码和文档差异，标注过期部分 | 每月 |

**Confluence 页面维护**：

在自动生成的文档页面顶部标注：

```html
<div style="background: #f0f0f0; padding: 8px; border-radius: 4px;">
  <strong>Auto-generated by AI</strong> |
  生成时间：2026-05-12 |
  基于 commit：abc1234 |
  <a href="#">重新生成</a>
</div>
```

### AI + Confluence 落地三步走

**第一步（第 1 周）**：AI 生成 Markdown -> 手动贴到 Confluence
- 零配置，立即可用
- 用 AI 扫描项目代码，生成所有 API 文档
- 一个人 1 天可以把主要模块的 API 文档补齐

**第二步（第 2-3 周）**：写自动化脚本，半自动发布
- 用 AI 帮你写 Python/Node.js 脚本
- 脚本读取 AI 生成的 Markdown，通过 Confluence API 发布
- 在 CI/CD 中加入"发版后自动更新文档"步骤

**第三步（第 4 周+）**：MCP Server 全自动
- 配置 Confluence MCP Server
- Claude Code 直接操作 Confluence
- 代码变更触发文档更新

### Confluence 上 AI 文档的标记规则

| 标签 | 含义 |
|------|------|
| `auto-generated` | AI 自动生成，未经人工审查 |
| `ai-reviewed` | AI 生成 + 人工审查 |
| `manually-written` | 纯人工编写 |
| `outdated` | 需要更新（AI 巡检标注） |

## 代码考古场景

当面对一个缺少文档的遗留代码仓时，`generate-docs` Skill 与 `code-archaeology` Skill 配合使用：

1. **先用 code-archaeology 理解代码**：全局扫描 → 模块深潜 → 提取业务规则
2. **再用 generate-docs 生成文档**：基于理解结果，生成标准化文档

典型流程：

```
# 对未知模块补齐文档的流程

## Step 1: 用 code-archaeology 分析模块
使用 code-archaeology Skill 的 Phase 2 对目标模块进行深潜分析，
产出业务规则表和数据流图。

## Step 2: 用 generate-docs 生成标准化文档
基于 Step 1 的分析结果，使用本 Skill 生成：
- 模块架构文档
- API 文档
- 数据模型文档

## Step 3: 标注置信度
对所有生成的内容标注置信度（参考 confidence-system）。
```

## 已有文档整合

当代码仓中已有部分文档（但可能过时或不完整）时：

### 整合策略

```
Prompt：
项目中已存在以下文档，请整合到新的文档体系中。

已有文档列表：
[列出已有文档路径和简要内容]

要求：
1. 读取每份已有文档的内容
2. 对照当前代码验证准确性
3. 保留仍然准确的内容，标注来源为"已有文档"
4. 更新过时的部分，标注变更内容
5. 补充缺失的部分，标注来源为"AI 分析"
6. 对每条内容标注置信度

输出：整合后的完整文档
```

### 差异处理

| 差异类型 | 处理方式 |
|----------|---------|
| 已有文档与代码一致 | 保留原文，标注 🟢 |
| 已有文档部分过时 | 更新过时部分，保留其余，标注 🟡 |
| 已有文档完全过时 | 基于代码重新生成，在文档中注明 |
| 已有文档描述的功能已不存在 | 标注为"已废弃"，不删除（保留历史） |
| AI 发现新功能但无文档 | 补充文档，标注为"AI 新增" |

## 置信度标注

生成的文档应使用 `confidence-system` 中定义的置信度体系进行标注：

| 标记 | 含义 | 在文档中的使用场景 |
|------|------|------------------|
| 🟢 高置信 | AI 从代码明确确认 | API 路径、参数类型等可直接从代码提取的信息 |
| 🟡 中置信 | AI 推断，需确认 | 业务规则、状态流转等需要理解业务上下文的信息 |
| 🔴 低置信 | AI 无法确定 | 复杂的异步逻辑、历史遗留的魔法数字等 |
| 🔵 人工确认 | 已有人验证 | 经团队确认后的内容 |

在生成文档时，对涉及业务逻辑、设计决策的部分添加置信度标注。详细规范参见 `docs/confidence-system.md`。

## 注意事项

- AI 生成的文档是**初稿**，必须人工审查后才能作为正式文档
- API 文档中容易遗漏：错误码、分页参数、权限要求
- 架构文档中容易出错：数据流方向、异步调用链、缓存策略
- 定期（每月）用 AI 巡检文档和代码的一致性（参考 `knowledge-sync` Skill）
- Confluence 上的自动生成页面要明确标注，避免误导读者
- 对未知代码仓，先使用 `code-archaeology` Skill 理解代码，再使用本 Skill 生成文档
