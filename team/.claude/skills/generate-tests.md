# Skill: 生成单元测试

> 用途：用 AI Agent 为现有代码自动生成单元测试，快速提升测试覆盖率。
> 这是阶段一"快速胜利"的核心 Skill。

## 触发条件

- 为旧代码补充测试
- 为新功能生成测试
- 提升模块测试覆盖率

## 输入要求

- **目标文件/模块**: 需要生成测试的代码路径
- **覆盖率目标**: 期望达到的覆盖率（默认 80%）
- **测试优先级**: 核心逻辑 > 边界情况 > 异常路径

## 执行步骤

### Step 1: 分析目标代码

1. 读取目标文件，理解：
   - 函数/方法的输入输出
   - 依赖项（数据库、外部API、其他模块）
   - 业务逻辑分支
   - 边界条件
2. 生成模块分析摘要

### Step 2: 识别测试场景

为每个函数/方法列出测试场景：

| 函数 | 场景 | 类型 | 优先级 |
|------|------|------|--------|
| [函数名] | 正常输入 | Happy path | P0 |
| [函数名] | 空值/null | 边界 | P1 |
| [函数名] | 非法输入 | 异常 | P1 |
| [函数名] | 并发场景 | 边界 | P2 |

### Step 3: 生成测试代码

测试代码规范：

- **测试框架**: 使用项目已有的测试框架
- **命名规范**: `test_[函数名]_should_[期望]_when_[条件]`
- **Mock 策略**: 外部依赖使用 Mock，内部逻辑尽量用真实调用
- **数据工厂**: 使用 Factory 模式创建测试数据，避免硬编码
- **独立性**: 每个测试用例独立运行，不依赖执行顺序

测试文件放置在对应的 `tests/` 目录下，与源文件结构镜像。

### Step 4: 运行并验证

1. 运行生成的测试，确保全部通过
2. 检查覆盖率报告
3. 标记未能覆盖的场景和原因

### Step 5: 输出报告

```
## 测试生成报告

### 模块: [模块名]
- 测试文件: [路径]
- 测试用例数: [数量]
- 覆盖率: [百分比]
- 覆盖的场景:
  - [x] 正常流程
  - [x] 边界条件
  - [ ] 异常场景（原因: [说明]）
- 后续建议: [需要人工补充的测试]
```

## 旧代码测试策略

为旧代码生成测试时，特别注意：

1. **不要修改源代码** - 先写测试理解行为，再考虑重构
2. **记录"意外行为"** - 如果测试揭示的代码行为与预期不符，记录下来
3. **使用快照测试** - 对于复杂输出，使用快照测试先锁定当前行为
4. **逐步提升** - 先覆盖主路径（P0），再扩展到边界和异常

## 注意事项

- AI 生成的测试需要人工审查业务逻辑正确性
- Mock 不能过度使用，否则测试失去价值
- 集成测试和 E2E 测试不适用此 Skill，需单独规划
