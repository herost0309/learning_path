# Skill: 新建 API 接口

> 用途：标准化的新 API 接口创建流程，确保一致性。

## 触发条件

当需要创建新的 API 接口时使用此 Skill。

## 输入要求

使用此 Skill 前，请提供以下信息：

- **接口路径**: 如 `/api/v1/users`
- **HTTP 方法**: GET / POST / PUT / DELETE
- **功能描述**: 接口的业务用途
- **请求参数**: 参数名、类型、是否必填、校验规则
- **响应格式**: 成功和失败的响应结构
- **权限要求**: 是否需要登录、需要的角色/权限

## 执行步骤

### Step 1: 编写 Spec

在 `docs/specs/` 目录下创建接口规格文件：

```
## [接口名称] Spec

### 基本信息
- 路径: [填写]
- 方法: [填写]
- 描述: [填写]

### 请求参数
| 参数 | 类型 | 必填 | 校验规则 |
|------|------|------|----------|
| [参数名] | [类型] | [是/否] | [规则] |

### 响应格式
成功 (200):
```json
{
  "code": 0,
  "message": "success",
  "data": { ... }
}
```

### 权限
- [填写权限要求]
```

### Step 2: 生成代码

根据 Spec 生成以下文件：

1. **路由定义** - 在路由文件中注册新接口
2. **Controller/Handler** - 请求处理逻辑
3. **参数校验** - 入参校验（使用项目校验框架）
4. **业务逻辑** - Service 层实现
5. **数据访问** - Repository/DAO 层（如需数据库操作）
6. **单元测试** - 覆盖正常流程和边界情况

### Step 3: 检查清单

- [ ] 参数校验完整（类型、范围、必填）
- [ ] 错误处理使用统一错误类
- [ ] 有权限校验（如需要）
- [ ] 数据库查询有索引覆盖
- [ ] 接口文档已更新
- [ ] 单元测试通过，覆盖率 ≥ 80%
- [ ] 无安全漏洞（XSS、SQL注入等）

### Step 4: 提交

提交信息格式：`feat: 新增 [接口名称] API`

## 注意事项

- 遵循 RESTful 规范
- 遵循项目已有的分层架构
- 复杂查询使用 Specification / Query Builder，不要拼接 SQL
