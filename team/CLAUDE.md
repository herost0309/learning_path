## 三、编码规范

### 3.1 通用规范

- 所有新代码必须有对应的单元测试，覆盖率目标 ≥ 80%
- 函数/方法不超过 50 行，超过则拆分
- 遵循项目已有的命名约定，不要混用风格
- 提交信息格式：`<type>: <description>`，type 包括：
  - `feat`: 新功能
  - `fix`: 修复Bug
  - `refactor`: 重构（不改变功能）
  - `test`: 测试相关
  - `docs`: 文档
  - `chore`: 构建/工具变更

### 3.2 后端规范

- API 接口遵循 RESTful 风格
- 错误处理统一使用项目封装的错误类，不要直接抛出原生异常
- 数据库查询必须有索引覆盖，N+1 问题必须在 Review 中拦截
- 新增 API 必须有参数校验和接口文档

### 3.3 前端规范

- 组件拆分粒度：单文件不超过 300 行
- 状态管理使用项目选定的方案（Redux/MobX/Pinia），不要混用
- 样式使用 [CSS Modules / Tailwind / styled-components]，保持一致
- 国际化：所有用户可见文本必须走 i18n

### 3.4 安全规范

- 禁止硬编码密钥、密码、Token，使用环境变量或密钥管理服务
- 所有外部输入必须校验和转义，防止 XSS/SQL 注入
- 敏感接口必须有权限校验
- 依赖包定期更新，CVE 漏洞必须在 48 小时内处理

## 四、AI 协作规范

### 4.1 Spec 驱动开发（SDD）流程

新功能开发遵循 **Spec → Agent实现 → 人工审查** 流程：

1. **写 Spec**: 开发者用自然语言编写需求规格（放在 `docs/specs/` 目录）
2. **Agent 实现**: 使用 Claude Code 或其他 AI Agent 根据 Spec 生成代码
3. **人工审查**: 开发者审查 Agent 生成的代码，确认功能和安全性
4. **测试验证**: 运行测试套件，确保通过

### 4.2 Agent 使用原则

- **人做决策，Agent 执行**: 架构设计、技术选型、安全决策由人做
- **Agent 生成代码必须 Review**: AI 生成的代码功能可能正确但缺乏架构判断力
- **优先使用 Skills**: 重复性工作使用 `.claude/skills/` 中已有的 Skill，没有则创建新的
- **保持 CLAUDE.md 更新**: 项目规范变更时同步更新本文件

### 4.3 旧代码处理原则

遵循 **"先映射，后修改"** 原则：

1. 先用 Agent 分析模块，生成模块地图（数据流、依赖关系、风险区域）
2. 先为旧代码生成测试（保护网）
3. 再逐步重构，每次改动后运行测试确认无回归
