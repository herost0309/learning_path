# 代码考古实操手册

> 写给技术负责人的实操指南。
> 场景：你接手了一个代码仓，文档残缺或完全没有。你需要在有限时间内用 AI Agent 快速补齐文档。

---

## 一、先搞清楚你的处境

在开始之前，花 30 分钟评估当前状态：

| 评估项 | 你的情况 | 说明 |
|--------|---------|------|
| 代码仓规模 | 小（<50 文件）/ 中（50-200）/ 大（>200） | 决定工作量和策略 |
| 已有文档情况 | 无 / 部分 / 过时 | 决定从零开始还是交叉比对 |
| 团队人数 | 1 人 / 2-3 人 / 更多 | 决定分工方式 |
| 可用时间 | 1 天 / 1 周 / 2 周 | 决定深度和范围 |
| 是否有了解系统的人 | 有 / 没有 | 决定验证策略 |

### 根据评估选择策略

| 情况 | 推荐策略 |
|------|---------|
| 小代码仓 + 无文档 + 1 天 | Phase 1 全局扫描 + 核心模块深潜，产出全景图和 API 文档 |
| 中代码仓 + 部分文档 + 1 周 | 完整三阶段流程，按天拆分 |
| 大代码仓 + 无文档 + 1 周 | Phase 1 全局扫描 + 按优先级深潜前 3 个核心模块 |
| 任意 + 2 周 | 完整三阶段 + 交叉验证 + 文档同步机制 |

## 二、一周实操计划（中型代码仓）

以下是针对中型代码仓（50-200 文件、5-10 个模块）的 5 天实操计划。

### Day 1：全局扫描，产出项目全景图

**目标**：对整个代码仓建立全局认知，产出全景图。

#### 上午：环境准备 + 初步扫描

```
操作步骤：
1. 确保能正常构建和运行项目
2. 用 code-archaeology Skill 的 Phase 1 逐步扫描
3. 先运行"项目全景图" Prompt
4. 再运行"入口点地图" Prompt
```

**使用 Prompt**：

```
分析当前项目的完整代码结构，生成项目全景图。
要求：
1. 识别技术栈（框架、语言版本、构建工具、主要依赖）
2. 列出目录结构及其职责说明
3. 列出所有业务模块
4. 识别项目类型（单体/微服务/Serverless/类库）
5. 估算代码规模
输出格式：Markdown
```

#### 下午：依赖分析 + 文档盘点

```
操作步骤：
1. 运行"依赖关系图" Prompt
2. 运行"现有文档盘点" Prompt
3. 将所有产出整理到一份文档中
```

**Day 1 产出**：
- 项目全景图（技术栈、目录结构、模块清单）
- 入口点地图（API 端点、定时任务、消息消费者）
- 依赖关系图（模块间、外部服务）
- 文档盘点表（已有 vs 缺失）

**检查点**：
- [ ] 能用一句话描述项目是做什么的
- [ ] 知道所有模块的名称和大致职责
- [ ] 知道项目依赖了哪些外部服务
- [ ] 知道哪些文档存在、哪些缺失

---

### Day 2：核心业务模块深潜（前 2 个模块）

**目标**：对最核心的业务模块进行深入分析。

#### 选择模块

根据以下标准选择今天要深潜的 2 个模块：

1. 直接涉及核心业务流程的模块
2. 代码量最大的模块
3. 其他模块依赖最多的模块

#### 对每个模块执行

**Step 1：代码与文档交叉比对**（如果已有文档）

```
Prompt：
分析 [模块路径] 的代码，并与以下已有文档交叉比对：
[粘贴已有文档或注明"无已有文档"]

要求：
1. 总结模块实际功能
2. 与已有文档的差异点
3. 代码中存在但文档未提及的功能
4. 标注差异的严重程度
```

**Step 2：提取业务规则**

```
Prompt：
深度分析 [模块路径] 的代码，提取业务规则。
重点关注：
1. if/else 条件分支的业务含义
2. 枚举值和常量的业务含义
3. 状态机逻辑
4. 校验规则
5. 计算逻辑
对每条规则标注置信度（高/中/低）。
```

**Step 3：追踪数据流**

```
Prompt：
追踪 [模块路径] 中的主要数据流。
对每个 API 端点，画出：请求入口 → 处理步骤 → 数据读写 → 响应
标注每一步的代码位置（文件:行号）。
```

**Day 2 产出**：
- 模块 1 的业务规则表（含置信度）
- 模块 1 的数据流图
- 模块 2 的业务规则表（含置信度）
- 模块 2 的数据流图

**检查点**：
- [ ] 知道这两个模块的核心业务逻辑
- [ ] 每条业务规则都有置信度标注
- [ ] 主要 API 的数据流已追踪清楚
- [ ] 不确定项已列出，附带了确认方式

---

### Day 3：剩余模块深潜 + 不确定项确认

**目标**：完成所有模块的深潜分析，确认 Day 2 标注的不确定项。

#### 上午：继续模块深潜

按优先级对剩余模块执行 Day 2 的同样流程。

#### 下午：确认不确定项

拿着 Day 2 产出的"不确定项清单"，找团队中了解系统的人确认：

```
确认方式：
1. 把不确定项清单发给有经验的同事
2. 每条不确定项标注：
   - AI 的推断是什么
   - AI 不确定的原因
   - 需要确认什么
3. 确认后更新置信度：🟡/🔴 → 🔵
```

**如果没有人可以确认**：

- 从 git blame / git log 找到相关代码的作者
- 看代码提交时的 commit message
- 看相关的需求文档或 JIRA/禅道记录
- 在文档中保留 🟡 标注，等后续有机会时确认

**Day 3 产出**：
- 所有模块的业务规则表
- 所有模块的数据流图
- 不确定项的确认结果

---

### Day 4：已有文档交叉比对 + 差异标注

**目标**：将 AI 分析结果与已有文档进行全面比对，产出差异报告。

#### 全面对比

```
Prompt：
将以下 AI 分析结果与已有文档进行完整对比。

AI 分析结果：
[粘贴所有模块的分析产出]

已有文档：
[列出所有已有文档的路径]

要求：
1. 逐份文档对比
2. 标注：一致的 / 有差异的 / AI 新发现的 / 文档提到但代码中没有的
3. 对差异项评估严重程度
4. 建议更新操作

输出：差异报告
```

#### 产出差异报告

```markdown
# 文档差异报告

## 概要
| 指标 | 数量 |
|------|------|
| 已有文档数 | ... |
| 与代码一致的文档 | ... |
| 需要更新的文档 | ... |
| 需要删除的文档（描述的功能已不存在） | ... |
| 需要新建的文档 | ... |

## 差异详情
### docs/api/orders.md
| 差异类型 | 描述 | 严重程度 |
|----------|------|---------|
| 缺失 | 退款接口未在文档中 | 高 |
| 过时 | 订单状态枚举增加了 2 个值 | 中 |
| 错误 | 参数名已改变 | 高 |
```

**Day 4 产出**：
- 完整的文档差异报告
- 需要更新/新建/删除的文档清单

---

### Day 5：产出完整文档集 + 发布

**目标**：基于前三天的分析，产出最终文档并发布。

#### 上午：生成文档

使用 code-archaeology Skill Phase 3 的 Prompt 逐一生成：

| 文档类型 | 使用 Prompt | 输出位置 |
|----------|-----------|---------|
| 模块架构文档 | Phase 3.1 | `docs/architecture/[模块名].md` |
| 数据模型文档 | Phase 3.2 | `docs/architecture/data-model.md` |
| API 文档 | Phase 3.3 | `docs/api/[模块名].md` |
| 风险地图 | Phase 3.4 | `docs/architecture/risk-map.md` |

每份文档需包含：
- 置信度标注（按 confidence-system 规范）
- 生成时间和基于的 commit hash
- "不确定项"章节

#### 下午：Review + 发布

**Review 检查清单**：

- [ ] 每份文档的第一句话能让人理解模块做什么
- [ ] API 文档覆盖了所有入口点
- [ ] 数据模型文档覆盖了所有表/实体
- [ ] 低置信项有明确的确认建议
- [ ] 没有敏感信息（密钥、内部 IP）
- [ ] Mermaid 图能正确渲染

**发布方式**：

1. **代码仓内**：提交到 `docs/` 目录，MR 标注 `[AI-Generated]`
2. **Confluence**：使用 generate-docs Skill 的发布流程
3. **团队分享**：在团队会议中花 15 分钟过一遍全景图和关键发现

#### 发布后的文档维护

在项目中设置长期同步机制：

1. 添加 `.git/hooks/pre-commit`（参考 knowledge-sync Skill）
2. 设置月度巡检日历提醒
3. 在 MR 模板中添加文档检查项

**Day 5 产出**：
- 完整的文档集（架构、API、数据模型、风险地图）
- 文档已发布到目标位置（代码仓/Confluence）
- 长期同步机制已建立

## 三、如何处理"AI 不确定"的部分

### 常见的不确定场景

| 场景 | AI 的表现 | 处理方式 |
|------|---------|---------|
| 变量名是缩写 | AI 无法推断业务含义 | 查 git blame 找原作者确认 |
| 复杂的 if/else 嵌套 | AI 能分析代码但不确定业务意图 | 标注为 🟡，等人工确认 |
| 异步逻辑 | AI 可能遗漏回调/事件处理 | 追踪完整调用链后再标注 |
| 被注释掉的代码 | 不确定是废弃还是临时禁用 | 检查 git log 确认 |
| 硬编码的魔法数字 | AI 能识别但不能确定含义 | 标注为 🟡，建议提取为常量 |
| 第三方库的隐含行为 | AI 可能不了解库的内部机制 | 查库的文档确认 |

### 置信度提升路径

```
🔴 低置信（AI 无法确定）
  ↓ 找到相关 git 提交记录，查看 commit message
🟡 中置信（有线索但不确信）
  ↓ 问团队中有经验的人，或查需求文档
🟢 高置信（AI 从代码明确确认）
  ↓ 经人工验证确认无误
🔵 人工确认（最终状态）
```

### 找谁确认

| 确认对象 | 适用场景 | 效率 |
|----------|---------|------|
| 原模块开发者 | 业务逻辑、设计决策 | 最高（如果还在团队） |
| 技术负责人 | 架构决策、技术选型 | 高 |
| 产品经理 | 业务规则、需求背景 | 中（只懂业务不懂实现） |
| 测试人员 | 边界条件、异常场景 | 中（从测试用例反推） |
| Git 历史 | 代码变更原因、时间线 | 中（需要自己推断） |

## 四、常见问题

### Q: 代码仓太大，一周做不完怎么办？

按优先级分析核心模块，产出全景图 + 核心 2-3 个模块的文档。其余模块列入待办，后续按需补齐。

### Q: AI 分析结果不准确怎么办？

AI 的分析结果都需要人工 Review。重点关注：
- 业务规则的推断是否合理
- 数据流是否完整
- 低置信项是否标注到位

不准确的部分标注为 🟡 或 🔴，等后续确认。

### Q: 团队中没有人了解系统怎么办？

完全依赖 AI 分析 + Git 历史推断：
1. AI 分析代码逻辑（事实层面通常准确）
2. Git log/blame 了解变更历史和原因
3. 从代码测试用例反推业务预期
4. 所有推断都标注为 🟡 中置信
5. 在文档中明确标注"未经人工确认"

### Q: 已有文档很多但不确定是否准确？

优先做文档 → 代码反向校验（knowledge-sync Skill 的"反向校验"功能），先知道哪些文档准确、哪些过时，再决定是更新还是重写。

### Q: 如何让文档持续保持更新？

建立三个机制：
1. **Pre-commit 钩子**：代码变更时提醒检查文档
2. **月度巡检**：AI 自动扫描代码和文档差异
3. **MR 检查清单**：Review 时检查文档是否同步

详见 `knowledge-sync` Skill。

## 五、效果评估

完成代码考古后，用以下指标评估效果：

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| 文档覆盖率 | 核心模块 100%，其他模块 ≥ 60% | 模块数 × 文档完整性 |
| 高置信条目占比 | ≥ 70% | 🟢 + 🔵 条目数 / 总条目数 |
| API 文档完整度 | 100% 端点覆盖 | 入口点地图 vs API 文档 |
| 新人理解时间 | 从 2-3 天降到半天 | 让新人读文档后反馈 |
| 文档过时率 | 月度巡检过时文档 < 10% | 巡检报告 |
