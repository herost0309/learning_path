# 文档置信度体系说明

> 定义技术文档的质量标注体系，让团队能区分"AI 推断"和"人工确认"。
> 所有通过 AI 生成的技术文档都应遵循此体系进行标注。

---

## 一、置信度定义

### 四级标注

| 标记 | 名称 | 含义 | 适用场景 |
|------|------|------|---------|
| 🟢 | 高置信 | AI 从代码中明确确认 | 代码有清晰的命名、注释或类型定义，AI 可以直接对应 |
| 🟡 | 中置信 | AI 推断，需人工确认 | AI 根据代码逻辑推断，但缺少直接证据或命名不够清晰 |
| 🔴 | 低置信 | AI 无法确定 | 可能来自过时文档、代码逻辑不清晰、或业务含义依赖上下文 |
| 🔵 | 人工确认 | 已有人工验证 | 经过团队中有经验的人确认，信息准确可靠 |

### 标注原则

1. **默认标注**：AI 生成的所有内容初始标注为 🟢 或 🟡，不会出现 🔵
2. **宁可标低**：如果不确定，标注为更低的置信度而非更高
3. **必须标注**：文档中涉及业务规则、设计决策、数据含义的部分必须标注
4. **不标注纯事实**：代码结构、API 路径等纯事实描述无需标注置信度

## 二、Markdown 标注格式

### 2.1 内联标注

在行内文字后用括号标注：

```markdown
订单创建后状态默认为"待支付" 🟢
超时 30 分钟未支付自动取消 🟡 — 推断自 ScheduledTask 中的超时检查逻辑
退款后库存自动回补 🔴 — 代码中未找到明确的库存回补逻辑
订单金额包含运费 🔵 — 李四确认于 2026-05-10
```

### 2.2 表格标注

在表格中增加"置信度"列：

```markdown
## 订单状态流转

| 当前状态 | 触发条件 | 目标状态 | 置信度 |
|----------|---------|---------|--------|
| 待支付 | 支付成功回调 | 已支付 | 🟢 |
| 已支付 | 超时未发货 | 已取消 | 🟡 推断自超时逻辑 |
| 已支付 | 卖家发货 | 已发货 | 🟢 |
| 已发货 | 买家确认收货 | 已完成 | 🔵 2026-05-10 张三 |
| 已完成 | 买家申请退款 | 退款中 | 🔴 需确认退款流程 |
```

### 2.3 章节级标注

对整个章节标注置信度：

```markdown
## 三、折扣计算规则 🟡

> 本章内容由 AI 从 PricingEngine 类推断，计算规则需人工确认。

- 普通用户：无折扣
- VIP 用户：95 折
- SVIP 用户：9 折
- 大促期间：额外 9 折（可叠加）

推断依据：PricingEngine.calculate() 方法中的 discountRate 计算逻辑
```

### 2.4 不确定项汇总

在文档末尾汇总所有低置信项：

```markdown
---

## 不确定项清单

| 项目 | 置信度 | 位置 | 建议确认方式 |
|------|--------|------|-------------|
| 退款流程的状态流转 | 🔴 | 3.2 节 | 联系原支付模块开发者 |
| 大促折扣是否可叠加 | 🟡 | 三、折扣计算规则 | 确认产品需求文档 |
| 库存预留的超时时间 | 🟡 | 二、库存管理 | 检查 Redis TTL 配置 |
```

## 三、Confluence 标注格式

### 3.1 高置信内容

直接展示，无需特殊标注：

```html
<p>订单创建后数据写入 <code>orders</code> 表。</p>
```

### 3.2 中置信内容

使用黄色提示框：

```html
<ac:structured-macro ac:name="info">
  <ac:parameter ac:name="color">Yellow</ac:parameter>
  <ac:rich-text-body>
    <p><strong>AI 推断（需确认）</strong></p>
    <p>VIP 用户享受 95 折优惠。</p>
    <p><em>推断依据：PricingEngine 中的 discountRate 计算逻辑。</em></p>
    <p><em>如确认无误，请删除此提示框并将置信度标记为 🔵。</em></p>
  </ac:rich-text-body>
</ac:structured-macro>
```

### 3.3 低置信内容

使用红色警告框：

```html
<ac:structured-macro ac:name="warning">
  <ac:rich-text-body>
    <p><strong>不确定项</strong></p>
    <p>退款流程的状态流转路径不明确。</p>
    <p><em>请联系原模块负责人确认，或查看 git 历史记录。</em></p>
  </ac:rich-text-body>
</ac:structured-macro>
```

### 3.4 人工确认内容

使用绿色成功框：

```html
<ac:structured-macro ac:name="info">
  <ac:parameter ac:name="color">Green</ac:parameter>
  <ac:rich-text-body>
    <p><strong>人工确认</strong></p>
    <p>订单金额包含运费。</p>
    <p><em>确认人：张三 | 确认日期：2026-05-10</em></p>
  </ac:rich-text-body>
</ac:structured-macro>
```

## 四、置信度提升流程

### 4.1 从低到高的升级路径

```
    🔴 低置信（AI 无法确定）
         ↓
    找 git 历史、查需求文档、看测试用例
         ↓
    🟡 中置信（有线索但不确信）
         ↓
    问团队中有经验的人
         ↓
    🟢 高置信（AI 从代码确认）或直接到 🔵
         ↓
    经人工验证确认
         ↓
    🔵 人工确认（最终状态）
```

### 4.2 升级操作

每次置信度升级时，需要：

| 操作 | 说明 |
|------|------|
| 更新标记 | 将文档中的标记从 🟡 更新为 🔵 |
| 记录确认人 | 在标记旁注明确认人和确认日期 |
| 删除提示框 | Confluence 中的黄色/红色提示框改为绿色或删除 |
| 提交变更 | 将更新后的文档提交到代码仓或 Confluence |

### 4.3 确认记录格式

```markdown
## 确认记录

| 日期 | 文档 | 原置信度 | 新置信度 | 确认人 | 备注 |
|------|------|---------|---------|--------|------|
| 2026-05-10 | docs/api/orders.md | 🟡 | 🔵 | 张三 | 退款流程确认 |
| 2026-05-12 | docs/architecture/data-model.md | 🔴 | 🟡 | - | 查到部分 git 记录 |
```

## 五、文档审查流程

### 5.1 审查优先级

按置信度从低到高进行审查：

| 优先级 | 置信度 | 审查方式 | 时间投入 |
|--------|--------|---------|---------|
| P0 | 🔴 低置信 | 必须审查，逐条确认 | 每条约 5-10 分钟 |
| P1 | 🟡 中置信 | 重点审查，抽样确认 | 每条约 2-5 分钟 |
| P2 | 🟢 高置信 | 快速浏览，验证关键点 | 每份文档 5-10 分钟 |
| P3 | 🔵 人工确认 | 定期抽检 | 每月检查一次 |

### 5.2 审查清单

审查每份文档时检查：

- [ ] 高置信项：代码是否仍然对应（快速确认）
- [ ] 中置信项：AI 的推断是否合理
- [ ] 低置信项：能否找到确认来源
- [ ] 新增功能：是否已在文档中补充
- [ ] 删除功能：是否已在文档中标注
- [ ] 置信度标记：是否需要升级或降级

### 5.3 审查报告模板

```markdown
# 文档审查报告

## 审查范围
- 文档：[文档名称]
- 审查人：[姓名]
- 审查日期：[日期]

## 审查结果

| 章节 | 原置信度 | 审查结论 | 操作 |
|------|---------|---------|------|
| 3.2 状态流转 | 🟡 | 确认正确 | 升级为 🔵 |
| 4.1 折扣规则 | 🟡 | 发现差异 | 修改为实际规则，保持 🟡 |
| 5.3 退款流程 | 🔴 | 找到确认人 | 升级为 🔵 |

## 遗留问题
- [ ] [未解决的问题]
```

## 六、团队使用规范

### 6.1 文档生成时

- AI 生成文档时自动标注 🟢 和 🟡
- 不确定的部分标注为 🔴
- 不要为了"好看"而标注为更高级别

### 6.2 文档审查时

- 审查后确认的内容升级为 🔵
- 发现错误的内容标注为 🔴 并添加修正说明
- 审查记录保存在文档末尾

### 6.3 日常维护时

- 每月巡检时检查所有 🔴 项是否可以升级
- 代码变更时检查受影响文档的置信度是否需要调整
- 团队成员可以随时确认任意条目（升级为 🔵）

### 6.4 新人使用时

- 新人应优先阅读 🔵 和 🟢 标注的内容
- 🟡 标注的内容可作为参考但需注意可能有偏差
- 🔴 标注的内容不要作为决策依据

## 七、指标追踪

### 7.1 关键指标

| 指标 | 计算方式 | 目标 |
|------|---------|------|
| 高置信占比 | (🟢 + 🔵) / 总条目 | ≥ 70% |
| 人工确认占比 | 🔵 / 总条目 | ≥ 40% |
| 低置信占比 | 🔴 / 总条目 | ≤ 10% |
| 平均确认速度 | 从 🔴 到 🔵 的平均天数 | ≤ 30 天 |

### 7.2 月度报告

每月统计置信度分布变化：

```markdown
# 置信度月度报告 - 2026年5月

## 分布概览
| 置信度 | 条目数 | 占比 | 上月 | 趋势 |
|--------|--------|------|------|------|
| 🟢 高置信 | 45 | 35% | 40% | -5% |
| 🟡 中置信 | 30 | 23% | 30% | -7% |
| 🔴 低置信 | 13 | 10% | 15% | -5% |
| 🔵 人工确认 | 41 | 32% | 15% | +17% |

## 改进
- 人工确认条目占比从 15% 提升到 32%
- 低置信条目从 15% 降到 10%
- 新增确认记录 26 条
```
