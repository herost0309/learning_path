# 第一步行动清单（实操版）

> 这是技术负责人一个人就能执行的方案。
> 不依赖团队配合，不依赖预算审批，今天就开干。

---

## Day 1：装工具 + 跑通流程（1-2 小时）

### 1. 安装 AI 编码工具

**推荐：通义灵码（免费，零门槛）**

- VS Code：扩展商店搜索"通义灵码"，点击安装
- JetBrains：插件市场搜索"通义灵码"，点击安装
- 安装后登录（支付宝/钉钉扫码即可）

**验证**：打开任意代码文件，输入注释 `// 实现：[功能描述]`，看 AI 是否给出建议。能给出就说明装好了。

### 2. 跑通一个完整流程

选项目中最简单的一个函数（纯计算、无外部依赖的最好），让 AI 给它生成测试：

```
操作步骤：
1. 打开那个文件
2. 选中函数代码
3. 对 AI 说："为这个函数生成单元测试，覆盖正常输入、边界情况和异常输入"
4. 把生成的测试代码放到 tests/ 目录
5. 运行测试
6. 记录：从开始到测试全部通过，花了多久
```

**预期**：30-60 分钟完成首次全流程（含摸索时间）。

### 3. 记录基线数据（5 分钟）

| 指标 | 今日值 | 怎么测 |
|------|--------|--------|
| 项目测试覆盖率 | ______ % | 运行 `npm test -- --coverage` 或 `mvn test` |
| 今天这个函数手工写测试估计要多久 | ______ 分钟 | 凭经验估算 |
| AI 实际花了多久 | ______ 分钟 | 刚才记录的 |
| AI 生成的测试能直接用吗 | 是 / 否（需要改多少） | 实际评估 |

## Day 2-3：用 AI 生成 CLAUDE.md（2-3 小时）

### 核心操作：不要手写，让 AI 分析代码来生成

在项目根目录，用通义灵码的 Agent 模式（或 Claude Code）执行：

```
Prompt（直接复制使用）：

分析当前项目的代码，帮我生成 CLAUDE.md 文件。

具体步骤：
1. 扫描 src/ 或 app/ 目录，列出实际存在的模块和每个模块的职责
2. 读取 package.json / pom.xml / build.gradle / requirements.txt，确定技术栈
3. 随机阅读 5-10 个源文件，总结出项目实际使用的命名规范
   - 变量命名：camelCase / snake_case？
   - 函数命名：动词开头还是名词？
   - 文件命名：kebab-case / PascalCase / camelCase？
4. 找出项目中的错误处理方式（看 catch 块、错误类、响应格式）
5. 找出项目的测试框架和测试目录结构
6. 如果有 CI/CD 配置文件（.gitlab-ci.yml / Jenkinsfile / .github/workflows/），分析部署流程

根据以上信息生成 CLAUDE.md。不要给我模板，给我基于实际代码的分析结果。
```

**审查清单**：

- [ ] 技术栈描述是否和 package.json/pom.xml 一致
- [ ] 目录结构是否和实际代码目录匹配
- [ ] 命名规范是否总结了大多数代码的实际风格（不是你期望的风格）
- [ ] 常用命令是否是真实可执行的
- [ ] 没有 `[填写]` 占位符

## Day 4-5：用 AI 生成全部 Skill（2-3 小时）

### 核心操作：让 AI 根据 CLAUDE.md 批量生成 Skill

```
Prompt（直接复制使用）：

根据以下项目信息，帮我生成 Claude Code 的 Skill 文件。

项目信息：
[粘贴刚才生成的 CLAUDE.md 内容]

团队现状：
- 5-8 人后端团队，测试覆盖率低
- 使用 [GitLab/GitHub] 管理代码
- 使用 [JIRA/禅道] 管理 Bug
- 上线流程手动步骤多

需要生成以下 Skill 文件：

1. .claude/skills/new-api.md
   - 基于我们实际的 [框架名] 框架
   - 包含一个真实的 API 示例（比如"创建用户"接口），从 Spec 到完整代码
   - 包括路由定义、Controller、Service、Repository、单元测试

2. .claude/skills/fix-bug.md
   - 包含一个真实的 Bug 示例（比如"空指针异常"），展示完整修复流程

3. .claude/skills/generate-tests.md
   - 基于我们实际的 [测试框架]
   - 包含一个真实的函数和对应的生成测试示例

4. .claude/skills/code-review.md
   - 列出基于我们项目的常见问题模式

5. .claude/skills/deploy.md
   - 基于我们实际的 CI/CD 配置

6. .claude/skills/refactor-module.md
   - 包含一个模块分析报告的实际示例

7. .claude/skills/generate-docs.md
   - AI 文档生成，包含 Confluence 发布方案

每个 Skill 要求：
- 使用我们项目的实际技术栈和框架
- 包含真实的代码示例（不是 [填写] 占位符）
- 步骤清晰，可直接执行
- 包含"常见问题"部分
```

**审查重点**：

- [ ] 代码示例使用的是项目实际的框架和语法
- [ ] 测试示例使用的是项目实际的测试框架
- [ ] 部署步骤和项目 CI/CD 配置匹配
- [ ] 没有 `[填写]` 占位符（用真实示例替代）

## 第 2 周：自己实战（用实际任务验证）

### 实战任务选择

选 3 个你手头正在做的真实任务：

| 任务 | 对应 Skill | 预期产出 |
|------|-----------|---------|
| 给一个旧模块补测试 | generate-tests | 测试覆盖率从 [X]% -> [Y]% |

| 开发一个新接口 | new-api | 完整的 API（含测试和文档） |

## 第 3 周：影响团队（不靠开会，靠行动）

### 周一：在 Code Review 中展示 AI

不要说"大家来用 AI"，而是在你 Review 别人的 MR 时，附带 AI 检查结果：

```
Review 评论示例：

常规人工 Review 意见：
- 第 42 行：这个查询在循环里，有 N+1 问题
- 第 88 行：缺少空值判断

AI 额外发现：
- 第 15 行：这个 SQL 查询没有走索引（AI 通过分析 WHERE 条件发现）
- 第 67 行：这里有个潜在的并发问题（AI 分析了事务边界）

结论：AI 帮忙多抓了 2 个问题，但业务逻辑的正确性（第 55 行的金额计算规则）还是需要人看。
```

### 周二-周四：帮同事解决痛点

等谁抱怨"这段代码看不懂"或"这个 Bug 排了一天"时：

```
话术："我来试试用 AI 帮你看一下"
操作：当着他的面用 AI 分析，2-3 分钟出结果
关键：让他在自己最头疼的问题上体验 AI
```

### 周五：非正式分享

午饭时跟 2-3 个同事聊你这两周的数据：

```
"我用 AI 给订单模块生成了测试，覆盖率从 5% 到 45%，才花了 3 个小时。
 以前估计得写两天。不过生成的测试有大概 30% 需要手动改，不是全能直接用。"
```

**注意**：说真实数据，包括失败的部分。这样更可信。

## 如果团队积极性不高怎么办

**核心策略**：不要推广 AI，推广"省时间"。

| 情况 | 做法 |
|------|------|
| 没人愿意试 | 不要求他们试。你继续用，在 MR 和文档中展示效果。 |
| 有人觉得 AI 不靠谱 | 承认不足，展示哪些场景靠谱哪些不靠谱。 |
| 有人担心被替代 | "AI 替代的是重复劳动，不是你。" |
| 领导不支持 | 先用免费工具，拿数据去申请。不申请先做事。 |
| 一个月后只有你一个人在用 | 没关系。你自己效率已经提升了。继续积累数据。 |

**终极策略**：你用 AI 省下的时间用来做架构优化，让你的代码质量和产出明显高于团队平均水平。这是最好的"无声推广"。
