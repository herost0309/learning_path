# 阶段一：快速胜利（第 1-4 周）—— 建立信心

> 目标：用 AI 解决最痛的问题，让团队感受到"有用"。
> 核心：不追求全面铺开，先让 3-5 人感受到效果，再扩散。
> 详细实操见 `docs/realistic-implementation.md`。

---

## 本阶段目标（基于公开数据的合理预期）

| 指标 | 目标值 | 行业参考数据 |
|------|--------|-------------|
| AI 代码采纳率 | ≥ 20% | 阿里初期 22%，GitHub Copilot 平均 46-55% |
| 试点模块测试覆盖率 | +30-40% | AI 生成测试效率约为手写的 5-8 倍 |
| 单人编码速度提升 | +20-30% | GitHub 研究：编码速度 +27%，任务完成率 +56% |
| 旧代码理解时间 | 减少 60-80% | AI 分析模块从 3-4 小时降至 20-40 分钟 |

---

## Week 1: 部署工具 + 试点启动

### 前提准备

- [ ] 确定 AI 先锋小组成员（3-5 人）
- [ ] 选择 AI 编码工具（建议：通义灵码 免费 / Claude Code 付费）
- [ ] 选定 1 个最痛的旧模块作为试点
- [ ] 安排 1 小时启动会议

### 启动会议议程（1 小时）

1. **为什么做这件事**（10 分钟）
   - 团队现状分析：人少事多、旧代码维护耗时
   - AI 工具不是替代人，是放大每个人的能力
   - 目标：减少重复劳动，让大家做更有价值的工作

2. **工具介绍与演示**（20 分钟）
   - 演示 AI 编码工具的核心功能
   - 演示"用 AI 给旧代码生成测试"的完整流程
   - 演示"用 AI 辅助 Code Review"

3. **试点模块讨论**（20 分钟）
   - 为什么选这个模块（维护频率高、Bug 多、团队最痛）
   - 试点方法：先 AI 生成测试 → 再 AI 辅助理解 → 再逐步重构
   - 分配任务：每人负责模块中 1-2 个文件

4. **约定与计划**（10 分钟）
   - 每周分享会时间
   - 试点期间的沟通方式
   - 第一周的任务分配

### Week 1 行动清单

| 谁 | 做什么 | 产出 |
|------|--------|------|
| 全员 | 安装 AI 编码工具 | 工具可用 |
| 全员 | 阅读 CLAUDE.md | 了解团队规范 |
| 先锋小组 | 用 AI 分析试点模块 | 模块分析报告 |
| 负责人 | 创建项目 CLAUDE.md | 规范文件就绪 |

## Week 2: 旧代码测试生成

### 重点任务

- [ ] 先锋小组成员用 `generate-tests` Skill 为试点模块生成单元测试
- [ ] 人工审查 AI 生成的测试，确认业务逻辑正确性
- [ ] 运行测试，记录覆盖率变化

### 执行方法

```
对试点模块中的每个文件：
1. 使用 AI Agent 分析代码
2. 使用 generate-tests Skill 生成测试
3. 人工 Review 测试的正确性
4. 运行测试，记录结果
5. 统计覆盖率变化
```

### Week 2 检查点

- [ ] 试点模块 ≥ 50% 的文件有了 AI 生成的测试
- [ ] 测试覆盖率从 [当前]% 提升到 [目标]%
- [ ] 团队成员对 AI 工具的使用熟悉度提升

---

## Week 3: AI 辅助 Code Review

### 重点任务

- [ ] 配置 AI Code Review 规则（使用 `code-review` Skill）
- [ ] 在 2-3 个 MR 中试用 AI Review
- [ ] 收集反馈，调整规则

### 配置要点

1. 安全规则：SQL 注入、XSS、敏感信息泄露
2. 性能规则：N+1 查询、全量查询
3. 规范规则：对照 CLAUDE.md 编码规范
4. 测试规则：检查测试覆盖

### Week 3 检查点

- [ ] AI Review 规则配置完成
- [ ] 至少 3 个 MR 使用了 AI Review
- [ ] 收集到 AI Review 的准确率反馈

## 关键成功因素

1. **选择"最痛"的模块** - 让效果显而易见
2. **先锋成员要积极** - 选择对新技术感兴趣的人
3. **不强制推广** - 先让效果说话，再自然扩散
4. **快速分享成果** - 不要等"完美"，有进展就分享
5. **容忍不完美** - AI 生成的测试/Review 可能不完美，关键是建立习惯

## 风险预案

| 风险 | 预案 |
|------|------|
| 先锋成员不积极 | 一对一沟通了解原因，必要时换人 |
| AI 工具效果不如预期 | 调整使用场景，找到最适合的切入点 |
| 团队焦虑"被替代" | 强调"AI 放大人"的定位，展示提升工作价值的案例 |
| 安全合规问题 | 确认 AI 工具的数据处理策略，必要时使用本地部署方案 |
