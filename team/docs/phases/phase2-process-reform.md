# 阶段二：流程改造（第 5-12 周）—— 释放效率

> 目标：将 AI 工具融入日常工作流，优化上线流程。
> 核心：从"个人用 AI"升级到"团队用 AI"。

---

## 本阶段目标

| 指标 | 目标值 |
|------|--------|
| AI 工具全团队使用率 | ≥ 80% |
| AI 代码采纳率 | ≥ 30% |
| Skills 库积累 | ≥ 5 个可复用 Skill |
| 上线流程中最耗时环节 | 缩短 ≥ 30% |
| 旧代码重构 | 完成 2-4 个模块 |

### 目标

建立团队可复用的 AI 工作流库。

### 行动清单

- [ ] 整理团队常见开发场景

## Week 7-8: Spec 驱动开发实践

### 目标

新功能开发采用 **Spec → Agent实现 → 人工审查** 流程。

### 实施步骤

1. **选择 1-2 个新功能** 作为 SDD 实践试点
2. **编写 Spec 文档** 放在 `docs/specs/` 目录
3. **使用 AI Agent 根据 Spec 生成代码**
4. **人工审查** 确认功能和安全性
5. **总结 SDD 流程的改进点**

### Spec 文档模板

```markdown
# [功能名称] Spec

## 背景
[为什么需要这个功能]

## 功能描述
[功能的详细描述]

## 接口设计
[API 接口定义]

## 数据模型
[数据表/模型设计]

## 非功能需求
- 性能: [要求]
- 安全: [要求]
- 兼容性: [要求]

## 验收标准
- [ ] [标准1]
- [ ] [标准2]

## 技术方案（可选，由 Agent 补充）
[实现方案细节]
```

### SDD 流程要点

```
开发者写 Spec → AI Agent 实现 → 开发者 Review → 运行测试 → 合并
     ↓                ↓               ↓            ↓
  需求明确        快速出代码       质量把关     安全保障
```

## Week 9-10: 旧代码重构启动

### 目标

用"先映射，后修改"方法，每月重构 1-2 个模块。

### 模块选择标准

| 因素 | 权重 | 说明 |
|------|------|------|
| 维护频率 | 高 | 经常需要改动的模块优先 |
| Bug 密度 | 高 | Bug 集中的模块优先 |
| 业务重要性 | 高 | 核心业务模块优先 |
| 重构难度 | 低 | 先易后难，积累信心 |

### 重构节奏

```
Week N:     选定模块 → Agent 生成模块地图 → 生成测试保护网
Week N+1:   执行重构（小步提交）→ 持续验证 → 重构报告
Week N+2:   观察期 → 收集反馈 → 准备下一个模块
```

### 参考方法

- **"先映射，后修改"** - Claude Code 的遗产代码处理方法
- **测试先行** - 先生成测试，再重构
- **Amazon Q 的 `/transform`** - 自动化升级转换的思路

## 关键成功因素

1. **Skills 必须实用** - 从团队实际痛点出发，不追求完美
2. **SDD 不强制** - 先在部分功能试点，效果好再推广
3. **重构不贪多** - 每月 1-2 个模块，持续进行比一次大改好
4. **MCP 逐步接入** - 先接最常用的工具，再扩展
5. **持续分享** - 保持每周分享会，展示新用法和效果

## 风险预案

| 风险 | 预案 |
|------|------|
| SDD 流程增加开发时间 | 初期可能慢，熟练后会加速；选对试点功能很重要 |
| 重构引入新 Bug | 测试保护网 + 小步提交 + 持续验证 |
| MCP 集成安全风险 | 最小权限原则，Agent 操作需审计日志 |
| 团队参与度下降 | 展示阶段性成果，让大家看到进步 |
