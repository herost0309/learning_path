## 一、先说实话：现状有多糟

拿一个典型的 5-8 人后端团队举例，以下是我见过最多的真实数据：

### 典型团队画像

| 维度 | 数据 | 来源 |
|------|------|------|
| 团队规模 | 5-8 人（实际写代码 4-6 人） | 多个中小团队实际调研 |
| 每日旧代码维护时间 | 40-60%（约 3-4 小时/天） | 开发者自报 |
| 测试覆盖率 | < 20%，核心模块接近 0% | SonarQube 扫描 |
| 每月线上 Bug 数 | 10-20 个，其中 2-3 个 P0/P1 | JIRA/禅道统计 |

| 技术文档 | 几乎没有，有也过时了 | Confluence 最后更新时间 |
| AI 工具使用 | 个别人私下用过，没有组织级推进 | 问卷 |

**核心矛盾**：想改善 -> 没时间改善 -> 继续在烂代码上堆代码 -> 更烂。

不是 PPT 数据，是各公司公开报告和团队实测的数据：

- AI 生成代码质量需要人工把控（churn rate 更高）
- 工具本身不是瓶颈，**人**才是

## 二、一个人怎么搞定：核心策略

### 策略：先"造好工具"，再"让大家用"

核心思路不是"说服大家用 AI"，而是：

1. **你自己先把所有 Skill 写好**（用 AI 帮你写）
2. **你自己在实际工作中用出效果**
3. **把效果数据摆在大家面前**
4. **给大家一个"开箱即用"的工具包，降低门槛到接近零**

```
传统思路（失败率高）：
  开会动员 -> 培训 -> 期望大家自觉使用 -> 逐渐放弃

我的思路（一个人驱动）：
  我先做好 -> 我先用好 -> 拿数据说话 -> 给大家"傻瓜式"工具
```

### 你的角色转变

```
以前：写代码的人
现在：写 Spec 的人 + 审查代码的人
关键：你不是在"推广 AI"，你是在"提升自己的产出"，其他人只是顺势加入
```

#### Day 3-4：编写 CLAUDE.md

**不要从模板开始**，从你的实际项目开始：

```bash
# 用 AI 帮你生成 CLAUDE.md
# 在项目根目录执行以下 prompt（给通义灵码或 Claude Code）：
```

**Prompt 示例**：

```
请分析当前项目的代码结构，帮我生成 CLAUDE.md 文件。
要求：
1. 扫描 src/ 目录，列出实际存在的模块和它们的职责
2. 检查 package.json / pom.xml / requirements.txt 确定技术栈
3. 找出项目中已有的命名规范（看现有代码怎么命名的）
4. 找出项目中的错误处理方式（看现有代码怎么处理错误的）
5. 找出项目中的测试框架和测试目录结构
6. 根据以上信息生成 CLAUDE.md

不要给我模板，给我基于实际代码的分析结果。
```

**关键**：用 AI 分析实际代码来生成规范，而不是你手写模板。这样 CLAUDE.md 才反映真实情况。

**预期产出**：
- 一个基于实际项目代码分析的 CLAUDE.md
- 包含真实的技术栈、真实的目录结构、真实的命名规范
- 不包含 `[填写]` 占位符

#### Day 5：编写全部 Skill（用 AI 帮你写）

**关键操作**：你不需要手写每个 Skill。用 AI 生成 Skill：

```
请根据以下信息，帮我生成 6 个 Claude Code Skill 文件。

项目信息（来自 CLAUDE.md）：
[粘贴你生成的 CLAUDE.md 内容]

团队现状：
- 5-8 人后端团队
- 测试覆盖率 < 20%
- 使用 [GitLab/GitHub] 管理代码
- 使用 [JIRA/禅道] 管理 Bug

需要生成的 Skill：
1. new-api.md - 新建 API 接口（基于项目实际的框架和分层）
2. fix-bug.md - 修复 Bug
3. generate-tests.md - 生成单元测试（基于项目实际的测试框架）
4. code-review.md - Code Review
5. deploy.md - 发布部署（基于项目实际的部署方式）
6. refactor-module.md - 旧代码重构

每个 Skill 必须包含：
- 具体到我们项目技术栈的代码示例
- 不用 [填写] 占位符，给出真实可用的模板
- 包含一个完整的示例（比如一个真实的 API 接口从 Spec 到代码的全过程）
```

**预期耗时**：2-3 小时（包括 Review AI 生成的 Skill 并调整）。

**产出**：
- 6 个基于项目实际情况的 Skill 文件
- 每个 Skill 包含真实代码示例（不是占位符）
- 可直接给团队使用

**真实数据记录表**（这是你后面说服团队的关键）：

| 步骤 | 手工做多久 | AI 辅助多久 | 节省 |

> 注意：这是测试生成的效率提升。日常编码的效率提升一般在 20-30%。

### 第 3 周：让团队"无痛"加入

**目标**：不是培训，是让大家在日常工作里自然接触到 AI。

**核心原则**：不要说"大家来学 AI"，而是说"我搞了个工具，你试试，能省时间"。

#### 周一：在 Code Review 中展示 AI 的价值

- 你 Review 别人的 MR 时，附带一份 AI Review 报告
- 不要强制要求，只是"顺便也跑了下 AI 检查"
- 在评论中标注：`[AI 发现]` 和 `[人工发现]`，让大家看到差异

**示例**：

```
MR Review 评论：

[AI 发现] foo.js:42 - 潜在的 N+1 查询，在循环中调用了 getUserById()
[AI 发现] bar.js:118 - 未处理的 Promise rejection
[人工发现] baz.js:55 - 这个业务逻辑和需求文档不一致（AI 看不出来）

结论：AI 帮忙抓了 2 个技术问题，但业务逻辑还是需要人看。
```

> **效果**：大家看到 AI 能抓到人容易漏掉的技术问题，开始觉得"有用"。

#### 周二-周三：帮同事用 AI 解决一个实际问题

**策略**：等（或主动问）谁在抱怨"这段代码太难懂了" / "这个 Bug 排查了一天" / "又要写测试好烦"。

**话术**：

```
"我来试试用 AI 帮你看一下" / "我有个工具可以自动生成测试，要不试试？"
```

**操作**：

1. 当着他的面，用 AI 分析那段代码（2-3 分钟出结果）
2. 让他看 AI 的分析是否准确
3. 用 generate-tests Skill 生成测试（5-10 分钟）
4. 让他跑测试看效果

> **效果**：亲眼看到 > 听一百次分享。让他在自己最头疼的问题上体验 AI。

#### 周四：分享你的数据（非正式）

- 把第 2 周的数据给 2-3 个同事看


## 四、团队积极性不高的应对方案

### 常见消极反应及应对

| 反应 | 真实原因 | 应对方式 |
|------|---------|---------|
| "没时间学新东西" | 不是没时间，是没看到价值 | 不让他"学"，让他在你帮助下"用一次" |
| "AI 生成的代码不靠谱" | 确实有不靠谱的时候 | 承认这一点，展示哪些场景靠谱哪些不靠谱 |
| "这是不是要替代我们" | 对未知的恐惧 | "AI 替代的是重复劳动，不是你。你的价值在于业务理解和架构设计" |
| "我习惯了自己的方式" | 惰性 | 不要求改变习惯，只是多一个选项 |
| "领导不支持" | 确实可能有这个问题 | 先用免费工具做出效果，拿数据去争取支持 |

### 最有效的策略排序

1. **在别人的痛点上用 AI 帮他解决一次**（最有效，亲眼见 > 100 次培训）
2. **分享真实数据（包括失败的）**（建立信任）
3. **把工具准备好到"零门槛"**（降低阻力）
4. **不强制、不考核**（强制推行适得其反）
5. **争取一个小的管理支持**（哪怕只是"不反对"）

### 如果一个月后还是只有你一个人在用

**没关系**。你自己用 AI 已经提升了 20-30% 的效率。继续用，继续积累数据。组织变革不是一个月能完成的。

但你可以做一件事：**把你用 AI 的过程和产出公开**。

- 你的 MR 标注 `[AI-Assisted]`
- 你的测试覆盖率持续提升
- 你的 Bug 修复速度比别人快
- 你有时间做架构优化了

这些是"无声的推广"，比任何会议都有效。

## 六、用 AI 帮你生成文档

### 6.0 接手未知代码仓：代码考古

如果团队接手了一个缺少文档的代码仓（或者文档严重过时），在生成文档之前，需要先**理解代码**。这就是"代码考古"——用 AI Agent 逆向分析代码，提取业务逻辑和架构设计。

**典型场景**：
- 接手遗留系统，前任开发者已离职
- 代码仓文档严重缺失或过时
- 跨团队协作需要快速理解对方系统
- 新入职需要快速熟悉项目

**做法**：

1. **全局扫描**（1 天）：用 AI 扫描整个代码仓，产出项目全景图、入口点地图、依赖关系图
2. **模块深潜**（2-3 天）：对核心模块逐一分析，提取业务规则、数据流、状态流转
3. **知识产出**（1 天）：将分析结果转为可维护的文档集

详细操作步骤参见 `docs/code-archaeology-guide.md`（代码考古实操手册），配合 `.claude/skills/code-archaeology.md` Skill 使用。

生成的文档使用**置信度体系**标注每条信息的可靠程度（🟢 高置信 / 🟡 中置信 / 🔴 低置信 / 🔵 人工确认），详见 `docs/confidence-system.md`。

文档完成后，使用 `knowledge-sync` Skill 建立长期同步机制，确保文档与代码持续一致。

### 6.1 为什么让 AI 生成文档

技术文档是团队最大的痛点之一：

| 问题 | AI 能帮什么 |
|------|------------|
| 没时间写文档 | AI 扫描代码自动生成 |
| 写了也过时 | 代码变了，AI 重新生成 |
| 格式不统一 | Skill 统一输出格式 |
| 不想写 | 用 AI 写初稿，人做审查 |

### 6.2 AI 生成文档的实用场景

#### 场景 1：API 文档自动生成

```
Prompt：
分析 src/routes/ 目录下所有 API 路由定义，
生成 RESTful API 文档，输出到 docs/api/ 目录。
格式参考 OpenAPI 3.0 规范。

对每个接口列出：
- 请求方法和路径
- 请求参数（query/path/body）
- 响应格式（成功/失败）
- 认证要求
```

#### 场景 2：模块文档自动生成

```
Prompt：
分析 src/modules/[模块名]/ 目录下的代码，
生成模块文档，包含：
1. 模块职责描述
2. 对外接口列表
3. 数据流图
4. 依赖关系
5. 已知问题和待优化点

输出到 docs/architecture/[模块名].md
```

#### 场景 3：变更日志自动生成

```
Prompt：
根据最近一次 release 分支的所有 commit，
生成 CHANGELOG 条目。
按 feat / fix / refactor / breaking change 分类。
```

### 6.3 在 Confluence 上实践

**现实问题**：大多数团队用 Confluence 做文档，但 AI 不能直接写 Confluence 页面。

**解决方案**：

#### 方案 A：AI 生成 Markdown + 手动贴到 Confluence（最简单）

```
流程：
1. AI 生成 Markdown 格式文档
2. 在 Confluence 新建页面，粘贴内容
3. Confluence 会自动将 Markdown 转为富文本

优点：零成本，立即可用
缺点：需要手动操作
适合：文档更新不频繁的场景
```

#### 方案 B：使用 Confluence API 自动发布（推荐）

```python
# 用 Python 脚本 + AI 生成的文档内容，通过 API 发布到 Confluence
# 这个脚本可以用 Claude Code 帮你生成

import requests

# 1. AI 生成文档内容（Markdown）
markdown_content = """
# API 文档
...
"""

# 2. 转换为 Confluence Storage Format（HTML）
# Confluence API 接受 storage format
storage_format = markdown_to_confluence_storage(markdown_content)

# 3. 通过 API 创建/更新页面
url = "https://your-company.atlassian.net/wiki/rest/api/content"
headers = {"Authorization": "Bearer YOUR_TOKEN"}
payload = {
    "type": "page",
    "title": "API Documentation - Auto Generated",
    "space": {"key": "DEV"},
    "body": {"storage": {"value": storage_format, "representation": "storage"}}
}
response = requests.post(url, json=payload, headers=headers)
```

#### 方案 C：用 MCP Server 连接 Confluence（最自动化）

在 Claude Code 的 MCP 配置中添加 Confluence Server：

```json
{
  "confluence": {
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-confluence"],
    "env": {
      "CONFLUENCE_URL": "https://your-company.atlassian.net/wiki",
      "CONFLUENCE_USERNAME": "your-email",
      "CONFLUENCE_API_TOKEN": "your-token"
    }
  }
}
```

配置后，Claude Code 可以：
- 读取 Confluence 页面（搜索已有文档）
- 创建/更新 Confluence 页面（直接发布）
- 在 Review 代码时自动关联相关文档

#### Confluence 上的文档模板

在 Confluence 空间中创建以下模板页面：

```
空间：DEV（开发团队）

页面结构：
├── 项目概览（首页，链接到各子页面）
├── API 文档/
│   ├── 模板：API 接口文档
│   └── [各模块 API 文档]
├── 架构文档/
│   ├── 模板：模块架构文档
│   └── [各模块架构文档]
├── 运维手册/
│   ├── 部署流程
│   └── 故障处理手册
└── AI 实践/
    ├── 每周 AI 使用报告
    ├── 最佳实践集
    └── 踩坑记录
```

#### 实操建议

1. **第一周**：用方案 A（AI 生成 + 手动贴），最快落地
2. **第二周**：用 AI 帮你写方案 B 的 Python 脚本，实现半自动
3. **第三-四周**：如有 Claude Code，配置方案 C 的 MCP Server

## 七、风险清单（不说废话版）

| 风险 | 概率 | 影响 | 实际应对 |
|------|------|------|---------|
| 团队完全不感兴趣 | 高 | 中 | 你自己用，不强迫别人。用产出说话。 |
| AI 生成的代码有 Bug | 高 | 中 | 所有 AI 代码必须 Review。AI 是助手不是作者。 |
| 公司不允许代码外发 | 中 | 高 | 通义灵码有企业版可私有化部署。确认后再用。 |
| AI 工具不稳定/限流 | 中 | 低 | 同时准备 2 个工具。通义灵码 + GitHub Copilot。 |
| 管理层不支持 | 中 | 高 | 先用免费工具，拿数据申请。不要先申请再做事。 |
| 一个月看不到效果 | 低 | 高 | 如果你自己用都没效果，重新评估工具选型。 |

## 八、关键原则

1. **一个人也能做** — 不需要等团队准备好
2. **用免费工具** — 不需要等预算审批
3. **用数据说话** — 不需要说服任何人
4. **在别人痛点上帮助** — 不需要培训
5. **接受不完美** — AI 不完美，但比没有好
6. **持续做** — 一个月不够就两个月，关键是开始
